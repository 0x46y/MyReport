# OCCT/WASM CAD修復検証の見せ方メモ

このメモは公開記事ではなく、ポートフォリオ化するときの編集方針として残す。

## 方針

- 過去に手元で検討用につけていた固有名は、公開記事やポートフォリオでは出さない。
- 将来ポートフォリオ作品として出す場合も、過去の検証コードをそのまま見せる印象にしない。
- 見せる場合は、目的、UI、サンプルデータ、説明文、公開範囲を整理し直した「現在の意図に合わせた技術実証」として扱う。

## 公開時の位置づけ

完成したCAD製品ではなく、以下のような技術実証として見せる。

- OCCTとWebAssemblyを使ったCADデータ修復の検証
- 壊れたSTL/IGESの診断・修復可能性の検証
- 商用CADの外側で、自由エッジ、穴候補、体積異常、ソリッド化失敗などを可視化する試み
- Web技術でCADカーネルに近い層を扱った実装経験

## 盛りすぎないこと

公開時には次のような書き方を避ける。

- CATIAを代替できると断言する
- すべての壊れたSTLを修復できると書く
- STEP変換が常に高品質にできると書く
- 商用CADのネイティブ形式を自由に解析できるように見せる
- OCCTを使えばすべて解決するように書く
- 過去の検証コードをそのまま現在の公開作品として見せる

## 説明文の方向性

公開するなら、次の方向に寄せる。

```text
OCCT(Open CASCADE Technology)をWebAssemblyでブラウザ上から扱い、STL/IGESデータの読み込み、表示、自由エッジ診断、穴候補検出、修復処理、STEP出力を検証した技術実証です。

商用CADから出力された壊れたSTLや、面・頂点情報が欠損したデータを対象に、CADソフトの外側でどこまで形状を診断・修復できるかを試しました。

完成したCAD製品ではなく、製造業の3Dデータ自動化におけるファイル変換、形状修復、WebAssembly活用、CADカーネル制御の検証プロジェクトです。
```

## 見せる画面候補

- STLまたはIGESを読み込んだ3D表示
- 自由エッジや穴候補の可視化
- 修復前後の比較
- Summaryや診断ログ
- STEP出力や変換結果
- 失敗ケースのログ

成功画面だけでなく、失敗や制約も見せてよい。CADデータ修復では「どこで壊れるかを調べた」こと自体にも価値がある。

## 技術スタック候補

```text
OCCT, WebAssembly, TypeScript, Three.js, React / Next.js, Rust
```

## 作品ページにする場合の構成

1. 何を作ったか
2. なぜ作ったか
3. 技術スタック
4. 試したこと
5. できたこと
6. 難しかったこと
7. 今後やるならどうするか

特に「難しかったこと」を入れる。

```text
WebAssembly版OCCTでは、必要な関数がバインディングに公開されていないことがあり、BRepの読み書きや体積計算で制約があった。
また、壊れたSTLに対して穴を塞いでも、ソリッド化で面が崩れる、体積が異常に膨らむ、自由エッジが残るといった問題があり、単純な変換処理では済まないことが分かった。
```

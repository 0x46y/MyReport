開発、クラウド、CADの記事で使う用語を、非エンジニアにも分かるように短く整理する。

この記事は辞書のように厳密な定義を網羅するものではなく、MyReportの記事を読むときに最低限つまずかないための用語メモとして使う。

## クラウド・料金まわり

### SKU

SKUは Stock Keeping Unit の略で、クラウドでは提供されるリソースの種類、サイズ、プランを区別する単位として使われることがある。

たとえば仮想マシンなら、CPU数、メモリ量、GPU有無、世代、リージョンによってSKUが分かれる。無料枠があっても、対象SKUの空きがなかったり、指定リージョンでは使えなかったりすることがある。

### リージョン

リージョンは、クラウド事業者のデータセンターがある地域の単位。

日本、米国、ヨーロッパなど、どの地域に置くかで、遅延、料金、利用できるサービス、法務・セキュリティ要件が変わることがある。

### アベイラビリティゾーン

アベイラビリティゾーンは、同じリージョン内にある独立した設備区画のようなもの。

障害に強い構成を作るために使う。ただし、リージョンやゾーンによって使えるリソースが違うことがある。

### 無料枠

無料枠は、一定量まで無料で使えるクラウドサービスの利用枠。

ただし、無料枠があることと、実際に自分の用途で問題なく使えることは別。対象サービス、対象リージョン、上限、期間、課金開始条件を確認する必要がある。

### 従量課金

従量課金は、使った分だけ料金が発生する課金方式。

少量の検証では安く始めやすいが、アクセス数、保存容量、API呼び出し回数、転送量が増えると料金も増える。

### egress

egressは、クラウドやストレージから外へ出ていくデータ転送のこと。

Web配信では、保存料金よりもegress料金が大きくなることがある。画像、動画、音声、AI生成物などを公開する場合は特に確認したい。

### Class A / Class B operations

Cloudflare R2などで使われる操作料金の分類。

Class Aは書き込みや一覧取得など、状態を変えたり管理したりする操作が中心。Class Bは読み取り系の操作が中心。保存容量だけでなく、何回読まれるか、何回書き込むかも料金に影響する。

### オブジェクトストレージ

オブジェクトストレージは、画像、動画、音声、バックアップ、ログ、AI生成物などのファイルを保存するためのストレージ。

Amazon S3、Cloudflare R2、Google Cloud Storage、Azure Blob Storage、Backblaze B2、Wasabiなどが代表例。フォルダ階層のように見えても、内部的にはオブジェクト単位で管理される。

### S3互換

S3互換は、Amazon S3と似たAPIで使えることを指す。

Backblaze B2やCloudflare R2のように、S3互換APIを用意しているサービスでは、S3向けのツールやライブラリを流用できる場合がある。ただし、完全に同じ機能が使えるとは限らない。

### IAM

IAMは Identity and Access Management の略で、誰がどのリソースに何をできるかを管理する仕組み。

クラウドでは、権限設定を誤ると情報漏えいや操作ミスにつながる。個人開発でも、公開してよいものと秘密にすべきものを分ける必要がある。

## Web・APIまわり

### README

READMEは、リポジトリの概要、使い方、セットアップ手順、注意点などを書く説明ファイル。

GitHubではリポジトリのトップに表示されることが多い。ただし、READMEだけで全ての実装詳細が分かるとは限らないため、必要に応じてソースコード、設定ファイル、テストも確認する。

### API

APIは、ソフトウェア同士が機能やデータをやり取りするための窓口。

外部サービスを自動化したり、別システムと連携したりするときに使う。ただし、APIでできることは提供側が決めるため、欲しい情報が必ず取れるとは限らない。

### SDK

SDKは Software Development Kit の略で、特定サービスや製品を使いやすくするための開発キット。

APIを直接呼ぶより簡単に扱えることが多いが、対応言語、バージョン、実行環境に制約がある場合がある。

### 依存関係

依存関係は、アプリやライブラリが動くために必要な別のソフトウェア、パッケージ、サービス、設定のこと。

npmやpipで入れるライブラリ、外部API、モデルファイル、OS固有のツールなどが含まれる。依存関係が変わると、同じ手順でも動かなくなることがある。

### Webhook

Webhookは、外部サービス側から自分のアプリへイベントを通知してもらう仕組み。

たとえば決済完了、ユーザー登録、ファイル更新などのイベントを受け取るときに使う。署名検証、ヘッダー、raw body、本番URLの確認が重要になる。

### OAuth

OAuthは、外部サービスのアカウントを使って認可するための仕組み。

Googleログインなどで使われる。リダイレクトURL、クライアントID、シークレット、本番ドメインの設定を正しく合わせる必要がある。

### raw body

raw bodyは、HTTPリクエストで送られてきた本文を、加工せずにそのまま扱ったもの。

Webhookの署名検証では、JSONとしてparseした後のデータではなく、元の本文が必要になることがある。

### Edge Runtime

Edge Runtimeは、ユーザーに近い場所でコードを実行するためのランタイム。

低遅延で配信しやすい一方、Node.jsと完全に同じAPIが使えるとは限らない。利用するライブラリがEdge Runtimeで動くか確認する必要がある。

### Node.js Runtime

Node.js Runtimeは、サーバー側でJavaScript/TypeScriptを動かす実行環境。

ファイルシステムや一部のNode.js APIを前提にしたライブラリは、Edge RuntimeやWorkers runtimeではそのまま動かないことがある。

### キャッシュ

キャッシュは、一度取得・生成したデータや結果を保存して、次回以降に再利用する仕組み。

Web配信では表示を速くするために使われる。重い変換処理やAI処理では、同じ処理を何度も実行しないためにも重要になる。

## DB・データ設計まわり

### SQLite

SQLiteは、サーバーを立てずにファイルベースで使える軽量なRDB。

ローカル検証、小さなツール、デスクトップアプリ、初期プロトタイプでは扱いやすい。一方で、複数人が同時に大量アクセスする公開Webサービスでは、PostgreSQLやMySQLのようなサーバー型DBを検討することが多い。

### RDB

RDBは Relational Database の略で、表形式のデータを関係として扱うデータベース。

PostgreSQL、MySQL、SQL Server、Oracleなどが代表例。厳密な整合性、トランザクション、複雑な検索、JOINが必要な業務データで候補になりやすい。

### PostgreSQL

PostgreSQLは、オープンソースのRDB。

トランザクション、制約、JOIN、拡張機能、JSON型などを扱える。SupabaseのようにPostgreSQLを基盤にしたサービスもある。

### Prisma

Prismaは、TypeScriptやJavaScriptからDBを扱いやすくするORM。

PostgreSQL、MySQL、SQLite、MongoDBなど複数のDBに対応している。ただし、DBごとの機能差が完全になくなるわけではない。

### ORM

ORMは Object Relational Mapping の略で、アプリのオブジェクトとDBのテーブルを対応づけて扱う仕組み。

SQLを直接書く量を減らし、アプリ側のDB操作を整理しやすくする。ただし、複雑なクエリやDB固有機能ではSQLやDB理解も必要になる。

### NoSQL

NoSQLは、RDB以外のデータベースを広く指す言葉。

FirestoreのようなドキュメントDB、キー-valueストア、グラフDBなどがある。SNS、チャット、画像投稿、ログ、柔軟なデータ構造を扱うプロトタイプでは使いやすい場合がある。

### Firebase

Firebaseは、Googleが提供するアプリ開発向けプラットフォーム。

Authentication、Firestore、Storage、Hostingなどを組み合わせて、認証付きアプリやプロトタイプを短時間で作りやすい。

### Firestore

Firestoreは、FirebaseやGoogle Cloudで使えるNoSQLのドキュメントDB。

ユーザーごとのデータ、チャット、投稿、設定のような柔軟なデータ構造を扱いやすい。読み書き回数やドキュメント設計は先に確認したい。

### MongoDB

MongoDBは、ドキュメント指向のNoSQLデータベース。

JSONに近い形でデータを扱いやすいが、内部的にはBSONという形式で保存される。柔軟なデータ構造に向く一方で、スキーマ設計を曖昧にしすぎると後から扱いにくくなる。

### Supabase

Supabaseは、PostgreSQLを中心に、認証、Storage、APIなどを組み合わせて使えるBaaS。

RDBを使いながらWebアプリに必要な認証やファイル保存もまとめて扱いやすい。無料枠、スリープ条件、本番環境との分離は確認したい。

### BaaS

BaaSは Backend as a Service の略で、認証、DB、Storage、APIなどバックエンド機能をサービスとして使う考え方。

開発速度を上げやすい一方で、利用するサービスの仕様、料金、制約に依存する。

### トランザクション

トランザクションは、複数の処理をひとまとまりとして扱い、全部成功するか全部失敗するかを管理する仕組み。

会計、在庫、注文、決済のように、途中まで成功して残りが失敗すると困る処理で重要になる。

### ロールバック

ロールバックは、処理に失敗したときに、変更前の状態へ戻すこと。

トランザクションと組み合わせて、途中までデータが更新されたまま残ることを防ぐ。

### JOIN

JOINは、複数の表を関連づけて検索するSQLの機能。

顧客、注文、商品、在庫のように、複数の表を組み合わせて正確に扱う業務では重要になる。

### RDS

RDSは Amazon Relational Database Service の略で、AWSが提供するマネージドRDBサービス。

新しい種類のDBというより、PostgreSQLやMySQLなどのRDB運用をAWSに任せやすくするサービス。バックアップ、監視、冗長化、月額費用は確認が必要。

### データウェアハウス

データウェアハウスは、過去のデータを集めて分析するための基盤。

アプリの現在状態を扱うDBとは目的が違う。売上分析、アクセス分析、ログ集計、機械学習用データ集計などで使われる。

### BigQuery

BigQueryは、Google Cloudが提供するデータウェアハウス。

大量データの分析や集計に向く。通常のアプリDBとして注文や在庫を更新する用途ではなく、分析基盤として考える。

### Snowflake

Snowflakeは、クラウド型のデータウェアハウス。

大量データの分析、部門横断のデータ活用、BI用途などで使われる。個人開発では最初から必要になることは少ない。

## 3D・CADまわり

### ジオメトリ

ジオメトリは、点、線、面、曲線、曲面、ソリッドなど、形そのものを表す情報。

3D表示では見た目が近ければよい場合もあるが、製造業のCADでは寸法や形状の正確さが重要になる。

### トポロジー

トポロジーは、点、エッジ、面、ソリッドがどのようにつながっているかを表す情報。

見た目が同じでも、面が閉じていない、エッジがつながっていない、ソリッドになっていない場合は、後工程で問題になることがある。

### B-Rep

B-Repは Boundary Representation の略で、境界表現と呼ばれる3D形状の表現方法。

面、エッジ、頂点などの境界情報でソリッド形状を表す。製造業CADやSTEPのような形式で重要になる。

### メッシュ

メッシュは、三角形や四角形などの小さな面の集まりで3D形状を表す方法。

STLは三角形メッシュで形状を表す。表示や3Dプリントには便利だが、設計履歴や厳密な曲面情報は落ちる。

### ソリッド

ソリッドは、中身が詰まった立体として扱える3Dデータ。

体積計算、干渉チェック、加工検討などでは、単なる表面ではなくソリッドとして閉じていることが重要になる。

### アセンブリ

アセンブリは、複数の部品を組み合わせた構造。

CADでは、部品同士の位置関係、親子関係、参照関係を持つ。ファイル変換でアセンブリ構造が失われると、後工程で扱いにくくなる。

### フィーチャー

フィーチャーは、穴、押し出し、面取り、リブなど、CAD上の設計操作や形状要素の単位。

STLなどに変換すると、フィーチャー履歴は基本的に残らない。

### パラメトリック

パラメトリックは、寸法や条件などのパラメータを変えることで形状を変更できる設計方法。

製造業CADでは重要だが、交換形式へ変換すると設計意図やパラメータが落ちることがある。

## 開発プロセスまわり

### PoC

PoCは Proof of Concept の略で、日本語では「概念実証」や「実現可能性の検証」と言われる。

完成品を作る工程ではなく、「この方法で本当に実現できるか」「どこに制約があるか」「本開発に進める前に何を確認すべきか」を小さく試す工程。

### 撤退条件

撤退条件は、調査やPoCを続けるか、保留するか、別案へ切り替えるかを判断するための基準。

「できなかったら終わり」という意味ではなく、費用、性能、ライセンス、運用、外部依存などがどの水準を超えたら現在の案を見直すかを先に決めておくための条件。

### アジャイル

アジャイルは、短い期間で作る、確認する、直す、次の範囲を決める、というサイクルを繰り返す開発の進め方。

最初にすべてを決めきれない開発や、実際に触りながら仕様を調整する必要がある開発に向いている。計画しないという意味ではなく、短い単位で計画と判断を繰り返す進め方。

### ウォーターフォール

ウォーターフォールは、要件定義、設計、実装、テスト、リリースのように、工程を順番に進める開発の進め方。

要件が明確で、後戻りが少なく、予算や期日を最初に見通したい場合に向いている。ウォーターフォール自体が悪いわけではなく、対象に合っているかが重要。

### スプリント

スプリントは、アジャイル開発で使われる短い作業期間のこと。1週間、2週間、1か月など、決めた期間で作業し、終わりに成果や課題を確認する。

### 検収

検収は、発注したものが合意した条件を満たしているかを確認し、受け取るかどうかを判断すること。

システム開発では、単に画面が動くかだけでなく、業務上必要な条件を満たしているかを確認する必要がある。

### 受け入れテスト

受け入れテストは、発注側や利用者側が、作られたシステムを実際の業務目線で確認するテスト。

開発側がテスト項目作成を支援することはできるが、「業務として問題ないか」を最終的に判断するには、業務を理解している人の確認が必要になる。

### レガシーシステム

レガシーシステムは、長い期間使われてきた古いシステムのこと。

古いこと自体が問題とは限らないが、担当者がいない、ドキュメントがない、修正しづらい、他システムとの依存関係が複雑、といった状態になると改修や刷新が難しくなる。

### 外部依存

外部依存は、自社や自分たちだけでは変更できないソフトウェア、サービス、API、ライブラリ、ベンダー製品などに依存している状態。

外部依存がある場合、自分たちのコードだけを直しても解決できない問題が発生することがある。

### 暗黙知

暗黙知は、手順書や仕様書には明文化されていないが、現場の経験や関係者の判断として共有されている知識。

業務システムや自動化では、人が普段判断している暗黙知を明文化しないと、AIやプログラムに任せる範囲を決めにくい。

### 責任分界点

責任分界点は、どこまでを誰が責任を持つのかを分ける境界。

クラウド、外部サービス、AI、ベンダー製品、社内運用が絡む場合、障害や判断ミスが起きたときに誰が対応するかを事前に整理する必要がある。

### 非機能要件

非機能要件は、機能そのものではなく、性能、処理時間、可用性、セキュリティ、運用性、保守性などに関する要件。

「変換できる」だけでなく、「何分以内に終わるか」「失敗時に復旧できるか」「本番運用できるか」といった観点が含まれる。

### チェックポイント

チェックポイントは、途中まで成功した状態や中間成果物を保存しておき、失敗時にそこから再開できるようにする考え方。

重いデータ変換、AI処理、ビルド、解析処理では、毎回最初からやり直さないために重要になる。

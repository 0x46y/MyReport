CADのライセンスやベンダーロックインは、単なる費用ではなく、自動化でどこまで制御できるかを決める前提条件になる。

製造業の3D自動化では、技術的にできるかどうかだけでなく、ライセンスや製品構成の問題が大きく関わる。

たとえば、あるCADではSTLやIGESは出力できるが、STEP出力や高度な変換機能は別ライセンスや追加モジュールが必要になる、ということがある。これは単に「メーカーが意地悪をしている」という話ではなく、CADメーカー側の開発コスト、サポートコスト、品質保証、事業モデルが関係している。

この記事では、CADのライセンスやベンダーロックインを、設計判断に影響するものとして整理する。

## ネイティブ形式はメーカーの競争力でもある

CATIAのCATPartやCATProductのようなネイティブ形式は、そのCAD製品の中核に近いデータ形式だ。

ネイティブ形式には、単なる形状だけではなく、設計履歴、フィーチャー、パラメータ、アセンブリ構造、属性情報、参照関係など、製品ごとの考え方が反映される。ここはCADメーカーにとって競争力そのものでもある。

そのため、ネイティブ形式が完全にオープンで、誰でも同じ精度で読み書きできる状態になっているとは限らない。外部ツールが対応していても、内部的にはライセンスされた変換エンジンや専用SDKを使っている場合がある。

この構造を理解していないと、「ファイルがあるのだから自由に読めるはず」「拡張子が分かるのだから変換できるはず」と考えてしまう。しかし実際には、ネイティブ形式を高品質に読み書きするには、そのCAD製品や公式API、商用コンバーターに依存することが多い。

## 交換形式の出力にもコストがかかる

STL、IGES、STEPのような交換形式は、他のソフトウェアへデータを渡すために使われる。

ただし、交換形式への出力も無料で当然にできるものとは限らない。CADメーカーから見ると、変換機能の実装にはかなりのコストがかかる。

- 複雑な形状を正しく変換する必要がある
- 他CADで読み込める品質にする必要がある
- 単位、座標系、面の向き、属性情報を扱う必要がある
- バージョン差や標準仕様の違いに対応する必要がある
- 変換に失敗したときのサポートが必要になる
- 顧客の業務停止につながるため品質保証が重い

STEPのような標準形式は便利だが、標準があることと、各CADがすべてを完全に同じように実装していることは別だ。どのAPに対応しているか、どの属性を出せるか、どのライセンスで使えるかは製品構成に依存する。

そのため、高度な変換機能や特定形式の出力が有料になるのは、メーカー側から見ると開発費やサポート費を回収するための自然な判断でもある。

## 利用側にとっては設計自由度の制約になる

一方で、利用企業側から見ると、ライセンスやネイティブ形式への依存は設計自由度の制約になる。

たとえば、自動化で次のようなことをしたい場合がある。

- 大量のCATPartを読み込んで特定の形状を検出したい
- CATProductの構成をたどって部品関係を解析したい
- STLに変換されたデータの穴や自由エッジを修復したい
- IGESやSTEPに変換した後の品質を自動検査したい
- 特定のパラメータや設計履歴を抽出したい

このとき、必要なライセンスがない、APIが公開されていない、ネイティブ形式を直接読めない、出力形式に情報が落ちる、といった制約があると、実装方針が大きく変わる。

問題は「高いか安いか」だけではない。自分たちがどこまで制御できるかが変わる。

## ベンダーロックインは悪とは限らないが、前提に入れる必要がある

ベンダーロックインという言葉は悪い意味で使われがちだが、商用CADの場合は単純に悪とは言い切れない。

CADメーカーは長年にわたって、幾何計算、UI、データ管理、図面、解析、変換、PLM連携などに投資している。企業がその製品を使うことで、高度な機能、安定性、サポート、実績を買っている面がある。

ただし、ロックインがあることを前提に設計しないと、自動化プロジェクトでは後から困る。

- その機能は今のライセンスで使えるのか
- 将来も同じバージョンを使い続けるのか
- 取引先や下請けも同じ形式を扱えるのか
- 別CADへ移行するときにデータを取り出せるのか
- APIの制限で自動化できない処理はないか
- メーカーのアップデートで既存マクロが壊れないか

これらを最初に確認しないと、後から「実装できると思っていたが、ライセンスやAPIの都合でできない」という話になりやすい。

## オープンソースは別の戦略で成り立っている

FreeCADやOCCTのようなオープンソースの技術は、商用CADとは違う戦略で成り立っている。

OCCTは、3D CADデータを扱うアプリケーション開発向けのオープンソースSDKとして、サーフェスやソリッドモデリング、可視化、データ交換などの機能を提供している。FreeCADはOCCTをベースにしたジオメトリエンジンを使うオープンソースのパラメトリック3Dモデラーだ。

参考:

- [Open CASCADE Technology - Open Source Core Technology](https://old.opencascade.com/content/open-source-core-technology)
- [FreeCAD features](https://www.freecad.org/features.php?lang=com)

オープンソースには、ソースコードを読める、改造できる、ライセンス費用を抑えやすい、特定ベンダーへの依存を弱めやすいという利点がある。

一方で、商用CADに比べると、自分たちで調査・ビルド・検証・保守する責任が増える。高度な機能がそのまま揃っているとは限らないし、特定の業務要件に合わせて作り込む必要がある。

つまり、商用CADとオープンソースは、どちらが絶対に正しいという話ではなく、コストをどこで払うかが違う。

| 方針 | 払うコスト |
| --- | --- |
| 商用CADに依存する | ライセンス費、ベンダー依存、API制約、製品ロードマップへの依存 |
| オープンソースや内製に寄せる | 調査、実装、検証、保守、幾何計算の理解、品質保証 |

## 自動化の前に確認したいこと

CAD自動化を始める前に、少なくとも次の点は確認したい。

| 確認項目 | 理由 |
| --- | --- |
| 現在のCADバージョン | APIや対応形式がバージョンで変わる |
| 利用中のライセンス構成 | STEP出力、変換、API利用などが可能か確認する |
| 入力形式と出力形式 | CATPart、CATProduct、STL、IGES、STEPで情報量が違う |
| 必須情報 | 形状だけでよいのか、履歴やパラメータも必要なのか |
| 自動化対象 | UI操作でよいのか、APIで直接触る必要があるのか |
| 代替案 | できない場合に、形式変更、手動工程、PoC除外が可能か |

ここを確認しないまま「自動化できます」と進めると、後からライセンスやデータ形式の壁にぶつかる。

## まとめ

CADのライセンスやベンダーロックインは、単なる費用の話ではない。

それは、自分たちがどのデータを読めるか、どの形式で出せるか、どのAPIを使えるか、どのレイヤーまで制御できるかに直結する。

商用CADは強力だが、製品として公開されている範囲の外には出にくい。オープンソースや内製化は自由度を増やすが、調査、実装、品質保証の責任も増える。

製造業の3D自動化では、技術選定の前に、ライセンス、データ形式、API、出力品質、将来の保守まで含めて考える必要がある。

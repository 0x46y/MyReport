結論から言うと、無料枠があること、実際に検証できること、本番利用できることは別問題だ。

クラウドやバックエンドサービスの無料枠は、個人開発や小さな検証（PoC）では非常に役に立つ。ただし、無料で試せる範囲だけを見て技術選定すると、後から「検証環境では動いたが、本番では費用や制限が合わない」という状態になりやすい。

無料枠、SKU、リージョン、従量課金、egress、PoC などの用語は [開発・クラウド・CADの記事で使う用語メモ](/notes/system-development-terms) に短く整理している。

## 無料枠で見るべきこと

無料枠を見るときは、単に「無料で使えるか」ではなく、次の観点に分けて確認したい。

| 観点 | 確認すること |
| --- | --- |
| 利用期限 | 期間限定の無料枠か、常時無料枠か |
| 利用量 | 何GB、何リクエスト、何時間まで使えるか |
| 対象プラン（SKU） | 対象のプランやリソースサイズが決まっているか |
| リージョン | 利用したい地域で無料対象のリソースを作れるか |
| スリープ | 使っていないと停止・一時停止されるか |
| 転送料 | 外向き通信やダウンロードで課金されるか |
| 商用利用 | 個人検証だけでなく業務・商用でも使えるか |
| 移行性 | 本番移行時に別プランや別サービスへ移しやすいか |

特に重要なのは、「無料枠の存在」と「検証が止まらず進むこと」を分けることだ。無料対象のリソースがあっても、空きがない、対象リージョンが限られる、作成時に別の有料リソースが必要になる、ということはあり得る。

## 利用期限を見る

無料枠には、期限付きのものと、一定量まで継続して無料のものがある。

たとえば、クラウド各社には新規アカウント向けの無料枠やクレジット、Always Free のような継続的な無料枠が用意されている場合がある。ただし、対象サービス、期間、上限、超過後の課金条件はサービスごとに異なる。

確認したいこと:

- 何日、何か月、何年使える無料枠か
- 無料期間が終わった後に自動で課金されるか
- 常時無料枠と期間限定クレジットが混ざっていないか
- 学習用の検証と本番運用で条件が変わらないか

無料期間中に検証するだけなら問題なくても、ポートフォリオや社内検証として長く公開する場合は、無料期間終了後の費用を先に見たい。

## 対象プラン（SKU）とリージョンを見る

Azureを少し試した範囲では、無料対象のVM系リソースを作成しようとしたときに、利用可能なリージョンやSKUが見つからず、検証を進めにくいことがあった。

これはAzure全体が使いにくいという話ではない。Microsoft系の業務基盤と組み合わせるなら、Azureを選ぶ理由は強い。ただし、無料枠で検証する場合は「無料対象のリソースがある」だけでなく、「自分が今使いたいリージョンで実際に作れるか」を確認する必要がある。

確認したいこと:

- 無料対象のSKUがどれか
- そのSKUが利用したいリージョンで作れるか
- リージョンを変えると遅延、法務、セキュリティ要件に影響しないか
- 無料枠ではなく少額課金にした方が検証が早いか

無料で使える会議室があっても、予約が取れなければ検証は進まない。それと同じで、無料枠は「制度としてあるか」と「今使えるか」を分けて見る必要がある。

## スリープや停止条件を見る

バックエンド機能をまとめて使えるサービス（BaaS）やホスティングサービスでは、無料枠のプロジェクトが一定期間使われないとスリープ、停止、一時停止されることがある。

たとえばSupabaseのFreeプランでは、一定期間利用されていないプロジェクトが一時停止される条件がある。個人開発や検証では便利だが、ポートフォリオやデモとして常に見せたいものでは注意が必要になる。

確認したいこと:

- 何日使わないと停止されるか
- 停止後に復活できるか
- 復活までにどれくらい時間がかかるか
- 停止中に外部公開ページやAPIがどう見えるか
- 有料プランにした場合、月額費用を回収できる用途か

無料枠で作ったデモが、会社への報告やポートフォリオ確認のタイミングで止まっていると、見せたい成果が伝わりにくい。長期公開するものは、無料枠のスリープ条件を必ず確認したい。

## 転送料を見る

無料枠で見落としやすいのが、外向き転送（egress）だ。

保存容量は無料でも、ユーザーがファイルをダウンロードしたり、画像や動画を何度も閲覧したりすると、転送料が効くことがある。オブジェクトストレージ、CDN、画像配信、AI生成物の公開では特に重要になる。

確認したいこと:

- 保存容量の無料枠はいくらか
- 読み取り回数、書き込み回数に上限があるか
- 外向き転送は無料か、有料か
- CDNやキャッシュを使った場合に料金が変わるか
- 無料枠を超えたときにどの単位で課金されるか

Cloudflare R2のように外向き転送を無料としているサービスもある。一方で、S3やCloud Storageのように、保存料金だけでなく転送量や周辺サービスも含めて見る必要があるサービスもある。

この点は [オブジェクトストレージ料金比較メモ: S3 / R2 / GCS / Azure Blob / B2 / Wasabi](/reports/object-storage-pricing-2026) にも整理している。

## 商用利用と利用規約を見る

無料枠は、個人検証では使いやすくても、業務利用や商用利用では条件が変わる場合がある。

たとえばVercelのHobbyプランは個人・非商用向けと説明されている。個人の実験や学習では使いやすいが、商用や業務用途ではPro以上の条件を確認する必要がある。

確認したいこと:

- 個人利用だけの無料枠か
- 商用利用できるか
- 社内検証や顧客向けデモで使ってよいか
- 利用規約上、データの種類に制約がないか
- 会社のセキュリティポリシーや契約条件に合うか

「無料で動く」ことと「業務で使ってよい」ことは別だ。会社の成果として見せる場合は、利用規約や商用利用条件も確認対象に入る。

## 本番相当で試せるかを見る

無料枠で作ったものが、ローカルや検証環境では動いても、本番環境では同じように動かないことがある。

たとえば、Webhook、OAuth、Stripe連携、メール送信、ファイルアップロード、Edge Runtimeのような機能は、本番URL、HTTPS、環境変数、ヘッダー、raw body、時刻、ドメイン設定が関係する。

確認したいこと:

- 本番と同じドメインやHTTPSで試せるか
- WebhookやOAuthのリダイレクトURLを本番相当で確認したか
- ローカルではなくデプロイ後の環境で動作確認したか
- 無料枠でもログを見て原因を追えるか
- 本番移行時にプラン変更だけで済むか、構成変更が必要か

この点は [ローカルで動いたのに本番で動かないのはなぜか](/reports/local-production-edge-runtime-differences) にも整理している。

## 依存関係と撤退条件を見る

無料枠は導入しやすい一方で、便利な機能を使うほどサービスへの依存も強くなる。

Firebase、Supabase、Vercel、Cloudflare、AWS、Azure、GCPのどれを使う場合でも、採用理由だけでなく、採用しなかった理由と撤退条件を残しておくと後から説明しやすい。

確認したいこと:

- データを外へ取り出せるか
- 別サービスへ移行できる形式か
- 無料枠終了後の月額費用はいくらか
- 料金が上がったときの代替案があるか
- そのサービスを扱える担当者が複数いるか
- 障害時にどこまで自分たちで調査できるか

無料枠は「入口」としては強い。しかし、入口の軽さだけで本番構成まで決めると、後から移行コストや運用コストが大きくなることがある。

## 自分ならどう進めるか

自分が個人開発や技術検証で進めるなら、次の順番で見る。

1. 無料枠で小さく試せるか
2. 無料枠の条件で検証が止まらないか
3. 本番相当のURLや環境で確認できるか
4. 無料枠を超えた場合の月額費用を見積もる
5. 移行できるデータ形式や代替案を確認する

個人開発では、まずCloudflare WorkersやVercelで小さく公開し、データベースが必要になった段階でSupabaseやFirebaseを検討することが多い。理由は、初期費用を抑えながら、実際に公開した状態で「本当にデータベースが必要か」「どの程度の読み書きが発生するか」「無料枠を超えたときに払える用途か」を判断しやすいからだ。

最初から大きなクラウド構成を組む必要はない。Cloudflare、Vercel、Firebase、Supabaseのようなサービスで小さく試す価値は大きい。ただし、長期公開、本番利用、業務利用に進めるなら、無料枠を超えた後の費用と運用を先に見る。

場合によっては、無料枠を探し続けるより、少額課金して確実に使える環境を作る方が調査効率は高い。技術検証では、金額だけでなく、調査に使う時間もコストとして扱いたい。

## まとめ

無料枠は、技術検証や個人開発の強い味方だ。しかし、無料枠があること、実際に検証できること、本番で使えることは分けて考える必要がある。

見るべきなのは、利用期限、SKU、リージョン、スリープ、転送料、商用利用、ログ、移行性、撤退条件だ。

無料枠を使う目的は、永遠に無料で運用することではなく、早い段階で成立条件と制約を確認することだと考える。

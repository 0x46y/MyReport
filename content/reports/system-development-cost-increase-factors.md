結論から言うと、システム開発の費用は「プログラムを書く時間」だけでは決まらない。

何を作るのか、誰が使うのか、データを誰が整えるのか、公開後に誰が保守するのか、クラウドや外部サービスの費用を誰が払うのかによって、必要な時間と費用は大きく変わる。

発注側にとって大事なのは、最初から正確な金額を当てることではなく、「どの条件で費用が増えるのか」「何を決めれば見積もりしやすくなるのか」を知っておくことだ。

この記事では、システム開発を依頼するときに知っておきたい開発費の考え方を、発注者向けに整理する。

## 発注者が不安になる理由

発注者にとって不安なのは、金額がまったく見えないまま相談することだ。

「値段の書いていない店に入る」ような不安や、「海外旅行中のスマホ料金」のように、あとから大きな請求が来るかもしれない不安に近い。

だからこそ、発注側には「正確な見積もり」以前に、まず大まかな規模感が必要になる。

確認したいこと:

- 数万円で済む話なのか
- 数十万円の開発なのか
- 数百万円以上を見た方がよいのか
- 月額費用や保守費用が発生するのか
- 追加要望が出た場合にどう扱うのか

開発側は「要件を聞かないと分からない」と言うことが多い。これは逃げているというより、条件によって本当に金額が変わるためだ。

そのため、発注側と開発側で最初にそろえたいのは、「この機能はいくら」と断定することではなく、金額の桁が変わる条件である。

## なぜ見積もりはすぐ決まらないのか

システム開発では、見た目が似ていても中身がまったく違うことがある。

たとえば「施設を検索できるサイト」を作る場合でも、次のように条件で大きく変わる。

| パターン | 内容 | 費用に影響する要素 |
| --- | --- | --- |
| 小さく始める場合 | 施設20件、ログインなし、CSV更新、検索だけ | 画面、検索、データ整理が中心 |
| 本格運用する場合 | 施設5000件、管理画面、ログイン、権限管理、添付ファイル、保守あり | データベース、認証、権限、更新フロー、保守まで必要 |

発注側から見ると、どちらも「施設を検索するサイト」に見える。

しかし開発側から見ると、確認することが違う。

小さく始める場合は、画面とデータ整備が中心になる。一方で、本格運用する場合は、ログイン、権限管理、更新画面、バックアップ、問い合わせ対応、障害時の対応まで考える必要がある。

家を建てるときに、部屋数、土地、構造、設備、耐震、外構を確認せずに金額を決めにくいのと同じで、システム開発でも、使い方や運用条件を確認しないと見積もりしにくい。

## まずは桁感を共有する

正確な見積もりを出す前でも、発注側と開発側で大まかな桁感を共有できると相談しやすい。

以下は実際の見積もりではなく、規模感をつかむための目安である。依頼先、地域、品質要件、契約形態、既存データの状態、保守範囲によって大きく変わる。

| 規模 | 期間の目安 | 金額の桁感 | 例 |
| --- | --- | --- | --- |
| 軽微な修正 | 数時間〜数日 | 数万円〜十数万円程度になることがある | 文言修正、小さな表示調整、既存機能の軽微な変更 |
| 小さなWebページ・静的サイト | 数日〜数週間 | 数万円〜数十万円程度になることがある | 会社紹介、告知ページ、簡単な情報掲載 |
| 小規模な検索サイト・業務ツール | 数週間〜1、2か月 | 数十万円〜数百万円程度になることがある | 一覧、検索、CSV更新、簡単な管理機能 |
| ログインや管理画面を含むシステム | 1か月〜数か月 | 百万円台以上になることがある | 認証、権限管理、更新画面、外部連携 |
| 業務の中核になるシステム | 数か月〜年単位 | 数百万円〜数千万円以上になることがある | 基幹業務、医療、金融、複雑な承認フロー、監査対応 |

ここで大事なのは、金額そのものよりも桁が変わる理由だ。

同じWebシステムでも、静的に表示するだけなら小さく始められる。一方で、ログイン、権限管理、外部連携、個人情報、監査、保守まで含めると、必要な確認作業が増える。

発注側と開発側で最初に共有したいのは、「数万円の相談なのか、数十万円なのか、数百万円以上を見た方がよいのか」という大まかな規模感である。そのうえで、どこまでを初期開発に含めるかを相談した方がよい。

## 発注者が見落としやすい作業

発注側から見ると、開発費は「プログラムを書く時間」に見えやすい。

しかし実際には、コードを書く前後にも多くの作業がある。

たとえば、小さな機能追加でも次のような作業が発生する。

| 作業 | 内容 |
| --- | --- |
| 仕様確認 | 何を実現したいのか、既存機能に影響しないか確認する |
| 調査 | 既存コード、データ、外部サービス、制約を確認する |
| 設計 | どこを変更するか、どのデータを使うか決める |
| 実装 | 実際にコードを書く |
| テスト | 想定通り動くか、壊れていないか確認する |
| 本番反映 | 公開環境へ反映し、動作確認する |
| ドキュメント | 使い方、変更点、注意点を残す |
| 調整 | 発注側からの確認結果を受けて修正する |

そのため、実装だけなら1時間に見える作業でも、確認、テスト、本番反映まで含めると数時間から数日になることがある。

発注側が知っておきたいのは、「コードを書く時間」だけでなく、「安全に反映して、後から運用できる状態にする時間」も費用に含まれるということだ。

## 時間単価や人月は目安として見る

開発費を考えるとき、1時間、1日、1週間、1か月のような作業単位で費用を考えることがある。

システム開発では、1人が1か月作業する量を「1人月」と呼ぶこともある。

ただし、発注者向けに大事なのは、専門的な単価の計算方法を細かく覚えることではない。

まずは次のように考えると分かりやすい。

- 軽微な修正なら、時間単位や日単位で考えられる場合がある
- 小さな機能追加なら、数日から数週間の作業になる場合がある
- 管理画面やログインを含むと、月単位で考える必要が出やすい
- 調査や小さな検証では、完成物ではなく作業期間に対して費用を決める場合がある

また、会社や開発チームに依頼する場合、費用には作業者の時間だけでなく、進行管理、レビュー、テスト、契約、機材、ソフトウェア、社会保険や諸経費などが含まれることがある。

これは、作業者の時給だけを払っているというより、プロジェクトを安全に進めるための体制や管理も含めて依頼していると考えると分かりやすい。

ただし、この部分を発注側がすべて理解している必要はない。重要なのは、見積もりに何が含まれていて、何が別費用なのかを確認することだ。

## 費用は分けて見る

システム開発の費用は、大きく分けると次のように整理できる。

| 種類 | 内容 |
| --- | --- |
| 初期開発費 | 画面、機能、データベース、管理画面、ログインなどを作る費用 |
| データ整備費 | Excel、CSV、既存資料、住所、画像、添付ファイルなどを整理する費用 |
| インフラ費 | サーバー、クラウド、ストレージ、ドメイン、外向き転送などの費用 |
| 外部サービス費 | 地図API、決済、AI API、メール配信、認証サービスなどの利用料 |
| 保守費 | 不具合対応、軽微な修正、セキュリティ更新、問い合わせ対応の費用 |
| 運用費 | 情報更新、データ確認、バックアップ、監視、障害時対応の費用 |

初期開発費だけを見ていると、公開後に必要な費用を見落としやすい。

特にクラウドやAI APIを使う場合は、最初は無料枠で動いていても、利用者数、データ量、API呼び出し回数が増えると月額費用が変わる。この点は [無料枠で検証するときに確認したいこと](/reports/free-tier-verification-checklist) や [クラウドのランニングコストとベンダーロックインを見落とさないために](/reports/cloud-running-cost-vendor-lockin) とも関係する。

## 費用が増えやすい機能

次の表は、機能ごとに費用へ影響しやすいポイントを整理したものだ。

金額は依頼先、契約形態、品質要件、既存データの状態で変わるため、ここでは固定額ではなく影響度として見る。

| 機能・要素 | 費用への影響 | 理由 |
| --- | --- | --- |
| 静的ページ | 低 | 文章や画像を表示するだけなら比較的単純 |
| 独自ドメイン | 低 | 年額費用とDNS設定が中心。ただし所有者管理は必要 |
| 検索・絞り込み | 低〜中 | データ量、検索条件、速度要件で変わる |
| 地図表示 | 中 | 住所整理、緯度経度、地図API、表示件数を確認する必要がある |
| 管理画面 | 中 | 入力、編集、削除、確認、権限、操作ミス対策が必要 |
| ログイン | 中 | 認証、パスワード管理、セッション、退会や権限を考える必要がある |
| 権限管理 | 中〜高 | 管理者、一般ユーザー、編集者などの役割ごとに制御が必要 |
| 外部API連携 | 高 | 相手サービスの仕様、制限、障害、料金、変更に影響される |
| 決済連携 | 高 | Stripeなどの連携、Webhook、返金、失敗時処理、監査が関係する |
| AI利用 | 中〜高 | 精度確認、API費用、誤判定時の扱い、人による確認が必要 |
| データ移行 | 高 | 既存データの形式、入力ゆれ、重複、欠損、変換作業が発生する |
| 添付ファイル・画像 | 中〜高 | ストレージ、容量、外向き転送、権限管理、バックアップが関係する |
| 保守契約 | 継続費 | 公開後の修正、問い合わせ、障害対応、更新作業が必要 |

発注側からは「少し項目を増やすだけ」「ログインを付けるだけ」に見えることがある。

しかし開発側では、データ構造、権限、テスト、エラー処理、運用手順まで変わる場合がある。費用が増える理由を説明するときは、機能名だけでなく、その裏側で必要になる確認作業も一緒に伝えたい。

## データ整備は見落とされやすい

Webシステムでは、画面よりもデータ整備に時間がかかることがある。

たとえば、Excelで管理している一覧をWebに載せる場合でも、次のような確認が必要になる。

- 列名がそろっているか
- 空欄や重複がないか
- 表記ゆれがないか
- 住所や電話番号の形式がそろっているか
- 公開してよい情報か
- 古い情報が混ざっていないか
- 誰が正しいデータだと確認するか

CSVやJSONのようなデータ形式を使えば、プログラムで扱いやすくなる場合がある。ただし、形式を変えるだけでデータの中身が正しくなるわけではない。

データの正しさを確認する作業は、業務側の判断が必要になることが多い。開発側だけでは、どの情報が最新で、どの情報を公開してよいかを判断できない場合がある。

## 管理画面は便利だが費用も増えやすい

管理画面があると、発注側で情報を更新しやすくなる。

一方で、管理画面を作ると次のような機能も必要になりやすい。

- ログイン
- 権限管理
- 入力チェック
- 更新履歴
- 削除確認
- 操作ミス対策
- 画像やファイルのアップロード
- 公開前確認
- バックアップ

そのため、最初から管理画面を作るべきかは慎重に判断したい。

小規模なサイトなら、最初はCSVやMarkdownで更新し、利用状況を見てから管理画面を作る方法もある。これは手を抜くためではなく、最初に本当に必要な運用を確認するための判断である。

## 外部サービス連携は相手側の制約も見る

地図API、決済、メール配信、AI API、OAuth認証、外部データ連携などを使う場合、費用は自分たちの開発工数だけでは決まらない。

確認したいこと:

- 外部サービスの利用料金
- 無料枠や上限
- 商用利用の可否
- APIの呼び出し制限
- 障害時の代替手段
- 仕様変更への対応
- 本番環境での動作確認

外部サービス連携は便利だが、相手側の料金、仕様、制限、障害の影響を受ける。

小さな検証では動いていても、本番導入では料金、契約、監査、障害対応まで確認する必要がある。この点は [小さな検証（PoC）が成功しても本番導入できない理由](/reports/poc-success-production-adoption-risks) ともつながる。

## AI利用は「作る費用」と「確認する費用」を分ける

AIを使うと、分類、要約、検索、文章生成、画像解析などを短時間で試せることがある。

ただし、AI利用では「AI APIを呼び出す機能を作る費用」と「出力結果を確認する費用」を分けて考えたい。

確認したいこと:

- どのくらいの精度が必要か
- 誤判定した場合に誰が確認するか
- 人による最終確認が必要か
- API利用料はどれくらい増えるか
- 入力データに機密情報を含むか
- 結果を保存するか
- 監査や説明責任が必要か

AIは実装だけなら速く試せる場合がある。

しかし業務で使うなら、精度、責任、費用、運用を確認する必要がある。この点は [AIがコードを書く時代に人間は何を確認するのか](/reports/ai-code-generation-human-checkpoints) とも関係する。

## 発注前に整理すると見積もりしやすいこと

発注前に次の項目を整理しておくと、見積もりや進め方を相談しやすくなる。

| 項目 | 確認したいこと |
| --- | --- |
| 必須機能 | 最初に必ず必要な機能は何か |
| 後回しにできる機能 | なくても初期公開できる機能は何か |
| 利用者数 | 何人くらいが使うか |
| データ件数 | 最初に扱う件数、将来増える件数 |
| 更新頻度 | 毎日、毎週、月1回など |
| 更新担当 | 誰が情報を更新するか |
| ログイン | 必要か、誰がログインするか |
| 権限管理 | 管理者と一般利用者を分ける必要があるか |
| 外部連携 | 地図、決済、AI、メールなどを使うか |
| 公開範囲 | 一般公開か、社内だけか |
| 保守 | 公開後の修正や問い合わせを誰が見るか |
| 月額費用の上限 | クラウドや外部サービス費用をどこまで許容するか |

この表を最初から完璧に埋める必要はない。

むしろ、未定の項目を明確にして、調査や小さな検証の対象にすることが大事だ。

## 費用を抑えるためにできること

費用を抑えるには、単に安い依頼先を探すだけではなく、開発範囲を整理することが重要になる。

費用を抑えやすい進め方:

- 最初に作る機能を絞る
- 必須機能と任意機能を分ける
- 管理画面を最初から作るか検討する
- 既存データを整理してから渡す
- 更新頻度を決める
- 無料枠や低コストなクラウドを使う
- ただし本番利用時の料金も確認する
- 保守範囲を決める
- 追加要望の扱いを決める

小さく始めることは、品質を下げることではない。

最初に価値が出る範囲へ絞り、利用状況を見ながら改善するための判断である。

## 見積もりを見るときの注意点

見積もりを見るときは、総額だけでなく、何が含まれていて、何が含まれていないかを確認したい。

確認したいこと:

- 要件定義や打ち合わせは含まれるか
- データ整備は含まれるか
- デザイン調整は含まれるか
- 管理画面は含まれるか
- テストはどこまで含まれるか
- 本番環境へのデプロイは含まれるか
- ドメインやクラウド費用は別か
- 保守費用は別か
- 追加要望はどう扱うか
- 検収後の修正はどこまで含まれるか

安く見える見積もりでも、データ整備、保守、クラウド費用、追加修正が別になっている場合がある。

逆に高く見える見積もりでも、運用設計、テスト、保守、ドキュメント、引き継ぎまで含んでいる場合がある。

金額だけでなく、含まれる範囲を見ることが重要だ。

## まとめ

システム開発の費用は、画面数だけでは決まらない。

ログイン、権限管理、データ整備、管理画面、外部API連携、AI利用、保守、クラウド費用、障害対応が入ると、必要な工数も運用費も変わる。

発注側がすべての技術を理解している必要はない。しかし、どの要素が費用に影響するのかを知っておくと、開発側と話しやすくなる。

「このシステムはいくらか」だけではなく、「何を初期開発に含めるか」「何を後回しにするか」「公開後の費用を誰が持つか」を整理することが、無理のないシステム開発につながると考える。

システム開発では、最初から完成形を完全に決めきれる場合と、実際に調べながら進めないと分からない場合がある。

まっさらな状態から作る新規開発で、業務要件が明確で、意思決定者も現場理解を持っていて、後戻りが少ない前提なら、ウォーターフォール型の進め方が合う場面はある。ウォーターフォールは、要件定義、設計、実装、テストのように工程を順番に進める方法だ。工程、予算、期日を最初に見通しやすく、発注側にとっても社内説明がしやすい。この記事で使う用語は [システム開発の記事で使う用語メモ](/notes/system-development-terms) にもまとめている。

一方で、現在の開発では、過去のコードや設定が積み上がった既存システムの修正や刷新も多い。他社製品に依存している。昔作った担当者やベンダーが現場を離れている。ドキュメントが足りない。発注側にも、受け入れテストを設計できるほど業務とシステムの対応関係を分かる人が残っていない。受け入れテストとは、発注側や利用者側が、作られたシステムを業務目線で確認するテストのことだ。こういう領域では、最初の見積もり時点で不確実性がかなり残る。

このような開発では、最初から「固定仕様で、いつまでに、いくらで、必ず完成するもの」として扱える範囲と、調査しながら判断する範囲を分けておきたい。そこを曖昧にすると、発注側、受注側、営業側、現場のエンジニアの認識がずれやすい。

自分がこの話で重視したいのは、誰かを責めることではない。あとで揉めないように、できること、できないこと、調査が必要なことを早めに共有し、合意したプロセスに沿って進めることだ。説明責任を果たしながら、発注側も開発側も納得できる形で判断していくための話として整理する。

また、契約、法務、労務、情報管理のような上位のルールを守ることは前提になる。会社ごとの文化や独自ルールはあってよいが、それは法律、契約、労使関係、情報セキュリティなどの基本的なルールを守った上に乗せるものだ。根性論や曖昧な期待で押し切るのではなく、必要な確認、合意、記録を残しながら進める必要がある。

## 実現可能性の検証は完成保証ではない

PoCは Proof of Concept の略で、日本語では「概念実証」や「実現可能性の検証」と言われる。完成品を小さく作る工程ではなく、「本当に実現できるか」「どこに制約があるか」を早めに確かめる工程だ。用語の詳しい説明は [システム開発の記事で使う用語メモ](/notes/system-development-terms) にまとめている。

実現できるか分からない部分は、本開発に入る前に小さく確認する必要がある。

たとえば既存システムの改修や刷新なら、次のようなことを確認する。

| 確認項目 | 見る理由 |
| --- | --- |
| 既存システムの仕様が分かるか | ドキュメントや担当者が残っていない場合がある |
| 外部製品や外部APIの制約がないか | 自社だけでは変更できない依存関係がある |
| 実データで動くか | サンプルでは動いても、本番データで例外が出ることがある |
| 性能が現実的か | 件数、データ量、連携先の応答時間で変わる |
| 受け入れテストができるか | 発注側に業務と期待結果を判断できる人が必要になる |
| 失敗時の扱いを決められるか | 除外、手動確認、代替案、段階リリースなどの判断が必要になる |

実現可能性の検証の結果、「できる」と分かることもある。一方で、「この方法ではできない」「別の方法なら可能性がある」「この機能は高コストなので今回は外した方がよい」と分かることもある。

つまり、実現可能性の検証の成果は完成品ではなく、判断材料だ。

## アジャイルに進める必要がある理由

アジャイルは、短い期間で作る、確認する、直す、次の範囲を決める、というサイクルを繰り返す開発の進め方だ。不確実性が高い開発では、トライアンドエラーを繰り返しながら進めるしかない場合がある。

最初にすべての仕様を固定してから作るのではなく、短いサイクルで次のような判断をする。

- どの既存機能なら安全に変えられるか
- どの外部依存なら制御できるか
- どのデータや業務パターンなら安定して処理できるか
- どの処理は自動化できるか
- どの処理は手動確認を残すべきか
- どこから先は追加費用や追加期間が必要か
- どの機能は今回の範囲から外すべきか

これは「計画しなくてよい」という意味ではない。むしろ逆で、探索しながら判断するための進め方を事前に決める必要がある。

アジャイルに進める場合でも、次のような合意が必要になる。スプリントは、1週間や2週間などの短い作業期間のことだ。

| 合意すること | 理由 |
| --- | --- |
| 1スプリントで確認すること | 何を検証したのか曖昧にしない |
| 完了条件 | 動いた、見えた、使える、業務に耐える、を区別する |
| 除外条件 | 実現困難な機能を無限に追い続けない |
| 判断タイミング | 継続、縮小、代替案、停止を決める場を作る |
| 誰が判断するか | 現場、発注側、営業、開発側の責任を分ける |
| 追加費用の扱い | 調査で増えた作業を無償対応にしない |

アジャイルは、責任を曖昧にする方法ではない。不確実性を可視化し、一定期間ごとに判断するための進め方だ。

## 完成条件の認識をそろえる

システム開発では、「何をもって完成とするか」を先にそろえることが重要になる。

発注側は、「業務で使える状態になること」を完成だと考える。受注側は、「合意した仕様の範囲を実装し、検収条件を満たすこと」を完成だと考える。営業側は、「提案時に説明した効果が出ること」を期待として伝えていることがある。現場のエンジニアは、「技術的に実現できる範囲」と「業務上必要な範囲」の差に挟まれる。

この認識がそろっていないと、次のようなすれ違いが起こりやすい。

- 調査が必要な作業なのに、完成保証つきの開発として扱われる
- 仕様にない処理が、当然できるものとして期待される
- 既存システムや既存データの問題が、開発側の責任として扱われる
- 発注側の意思決定が遅れても、スケジュール遅延だけが問題になる
- 営業時の説明と現場で可能な実装範囲がずれる
- 途中で分かった制約を共有しても、後から「最初にできると言った」と言われる

特に、既存システムや商用ソフトに依存する開発では、外部要因が多い。API、ライセンス、データ品質、既存DB、既存業務フロー、利用者の暗黙知などが絡むため、開発側だけで制御できないことがある。

だからこそ、強い言い方で牽制するよりも、最初に完成条件、検収条件、除外条件、判断タイミングを明文化しておく方が建設的だ。これは開発側を守るためだけではなく、発注側が予算や社内説明をしやすくするためにも必要になる。

## 過去の大型開発紛争から見えること

日本でも、大型システム開発が頓挫し、発注側とベンダー側で訴訟になった事例は報道されている。

たとえばNHKと日本IBMの営業基幹システム開発をめぐる訴訟では、NHK側が2025年2月に約54億7000万円の支払いを求めて提訴したと報道されている。一方で、日本IBM側は、開始後に現行システムの解析を実施した上で移行方針やスケジュールを確定する契約だった、という趣旨の見解を示したと報道されている。

参考:

- [NHK、システム開発めぐり日本IBMを提訴 - ITmedia NEWS](https://www.itmedia.co.jp/news/articles/2502/04/news169.html)
- [日本IBM、NHKシステムの開発中断について見解 - ITmedia NEWS](https://www.itmedia.co.jp/news/articles/2502/07/news182.html)

また、文化シヤッターと日本IBMのシステム開発をめぐる訴訟では、最高裁が2025年1月に文化シヤッター側の上告を棄却し、日本IBMに約20億円の支払いを命じた判決が確定したと報道されている。

参考:

- [文化シヤッターの損害賠償請求訴訟、日本IBMの20億円支払い確定 - 日刊鉄鋼新聞](https://www.japanmetaldaily.com/articles/-/231149)

ここで個別事件の責任をこの記事で断定するつもりはない。重要なのは、大型開発では「何が契約範囲だったのか」「どこまでが調査だったのか」「既存システムの解析結果を踏まえてどう変更すべきだったのか」「完成条件を誰がどこまで合意していたのか」が大きな争点になり得るということだ。

規模や業種が違っても、既存システムの刷新や不確実性の高い開発では、同じ構造が起こり得る。

私は法律の専門家ではないため、個別事件の法的評価はしない。ただ、開発現場の教訓としては、難しいと分かった時点で早めに共有し、継続、縮小、停止、代替案の判断を明確にすることが重要だと考える。「ここは実現できない」「この条件なら追加調査が必要」「このまま進めると納期や費用に影響する」と適切なタイミングで伝え、判断を求めることが、結果的に関係者全員を守る。

## 営業側とも前提をそろえたい

技術的に不確実性が高い案件では、営業側の説明も重要になる。

営業段階で「できます」と強く言い切ると、発注側は完成を前提に予算やスケジュールを組む。しかし現場では、実データ、外部依存、既存システム、業務フローを調べて初めて分かることがある。

そのため、営業側も次のように説明した方がよい。

- まず実現可能性を確認するフェーズが必要
- 実現可能性の検証の結果によって、実装範囲を調整する
- できること、できないこと、追加調査が必要なことを分ける
- 固定価格で約束しやすい範囲と、探索が必要な範囲を分ける
- 既存データ、既存システム、外部製品に起因する問題は、別途判断が必要
- 一定期間ごとに継続、縮小、代替案、停止を判断する

営業がこの前提を一緒に握ってくれると、発注側にも開発側にも説明しやすくなる。結果として、現場のエンジニアが後から無理な約束を背負うリスクも減る。

## 発注側とも一緒に判断したい

発注側にも、開発を進める上で一緒に判断してもらいたいことがある。

もちろん、技術実装は開発側の責任だ。しかし、業務要件、既存データ、検収条件、優先順位、判断タイミングは発注側が関わらないと決められない。

特に受け入れテストは、本来は発注側の業務理解が重要になる。開発側がテスト項目の作成を支援することはできるが、「その結果で業務上問題ないか」を最終的に判断するには、業務を知っている人の確認が必要になる。ここをすべて開発側に任せると、仕様通りに動いているかではなく、業務として正しいかの判断まで開発側に寄ってしまう。

発注側に協力してもらえると進めやすいのは、たとえば次のようなことだ。

- 実データを提供する
- 正常データと異常データの例を出す
- 何をもって成功とするかを決める
- どの失敗なら許容できるかを決める
- 現場の暗黙知を共有する
- スプリントごとの判断に参加する
- 追加費用や範囲変更の判断を先送りしない

発注側が早い段階から判断に参加してくれると、開発側との認識差を小さくできる。探索型の開発では、発注側と開発側が同じ情報を見ながら、一緒に範囲や優先順位を決めていく方がよい。

## 同じ前提で進められる関係を作りたい

このような話は、要求を断るためではなく、揉め事にならないようにするために必要だと考えている。事前に説明し、合意し、記録し、プロセスとして正しく進めることで、発注側も開発側も判断しやすくなる。

不確実性があるなら、不確実性があると伝える。できることはできると言う。できないことや追加調査が必要なことも、早めに伝える。メリットだけでなく、制約やリスクも共有する。その上で、どう進めるかを一緒に決めたい。

同じ課題意識を持っている発注側、営業側、エンジニアとは、むしろ一緒にやっていきたい。技術的に難しい領域でも、前提をそろえて進められるなら、現実的な成果に近づける。

## 最初に決めておきたいこと

不確実性が高い開発では、契約や進め方の段階で次の点を明確にした方がよい。

| 項目 | 決めること |
| --- | --- |
| フェーズ分け | 調査、実現可能性の検証、本開発、運用を分ける |
| 成果物 | 調査報告、検証コード、プロトタイプ、本番機能を区別する |
| 完了条件 | 何ができればそのフェーズを完了とするか |
| 除外条件 | 実現困難なものをどう扱うか |
| 判断会議 | 継続、縮小、停止、追加調査をいつ決めるか |
| 役割分担 | 外部依存、既存データ不備、業務判断を誰が確認し、誰が判断するか |
| 追加費用 | 想定外の調査や実装が出た場合の扱い |
| 検収条件 | 技術的成功と業務上の利用可能性をどう分けるか |

ここを曖昧にしたまま進めると、後から「完成していない」「仕様が違う」「追加費用は払えない」「最初から分かっていたはず」という話になりやすい。逆に、最初に合意しておけば、途中で制約が見つかったときにも冷静に次の判断をしやすい。

## まとめ

不確実性が高い開発では、実現可能性の検証とアジャイルな進め方が必要になる。

ただし、アジャイルは「何となく作りながら進める」という意味ではない。できること、できないこと、追加調査が必要なことを短いサイクルで確認し、範囲と優先順位を更新していく進め方だ。

特に既存システム刷新や外部製品に依存する開発では、既存コード、外部サービス、ライセンス、既存データ、既存業務、暗黙知が絡む。開発側だけで制御できない問題も多い。

だからこそ、発注側、受注側、営業側、現場のエンジニアが、探索型の開発であることを最初に共有する必要がある。固定仕様として約束しやすい範囲と、調査しながら判断する範囲を分けることで、無用なすれ違いを減らせる。

自分としては、こうした前提を共有できる相手と、現実的に成果を出せる進め方を作っていきたい。

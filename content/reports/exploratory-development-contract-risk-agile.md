製造業の3D自動化や、既存システムの刷新のような開発では、最初から完成形を完全に決めきれないことがある。

実データを触らないと分からない。既存システムを解析しないと分からない。外部ソフトのAPIを試さないと分からない。ライセンスやデータ形式の制約を確認しないと分からない。こういう領域では、最初の見積もり時点で不確実性がかなり残る。

このような開発を、最初から「固定仕様で、いつまでに、いくらで、必ず完成するもの」として扱うと、発注側、受注側、営業側、現場のエンジニアの認識がずれやすい。

## PoCは完成保証ではなく、実現可能性を調べる工程

PoCは、完成品を小さく作る工程ではない。

本来は、実現できるか分からない部分を早めに確認する工程だと思う。

たとえばCAD自動化なら、次のようなことを確認する。

| 確認項目 | 見る理由 |
| --- | --- |
| 必要な情報がAPIで取れるか | 画面上で見えていてもAPIから取れるとは限らない |
| 対象データを読み込めるか | ファイル形式、バージョン、破損、単位系で失敗する可能性がある |
| 出力形式が使えるか | ライセンスや製品構成で出力できないことがある |
| 処理時間が現実的か | 大きな3Dデータでは読み込みや変換が重い |
| 品質判定ができるか | 見た目だけでなく、ソリッド性や体積を見たい場合がある |
| 失敗時の扱いを決められるか | 除外、手動確認、別形式で再出力などの判断が必要になる |

PoCの結果、「できる」と分かることもある。一方で、「この方法ではできない」「別の方法なら可能性がある」「この機能は高コストなので今回は外した方がよい」と分かることもある。

つまり、PoCの成果は完成品ではなく、判断材料だ。

## アジャイルに進める必要がある理由

不確実性が高い開発では、トライアンドエラーを繰り返しながら進めるしかない場合がある。

最初にすべての仕様を固定してから作るのではなく、短いサイクルで次のような判断をする。

- どのデータなら安定して処理できるか
- どの形式なら変換できるか
- どのAPIなら必要な情報を取れるか
- どの処理は自動化できるか
- どの処理は手動確認を残すべきか
- どこから先は追加費用や追加期間が必要か
- どの機能は今回の範囲から外すべきか

これは「計画しなくてよい」という意味ではない。むしろ逆で、探索しながら判断するための進め方を事前に決める必要がある。

アジャイルに進める場合でも、次のような合意が必要になる。

| 合意すること | 理由 |
| --- | --- |
| 1スプリントで確認すること | 何を検証したのか曖昧にしない |
| 完了条件 | 動いた、見えた、使える、業務に耐える、を区別する |
| 除外条件 | 実現困難な機能を無限に追い続けない |
| 判断タイミング | 継続、縮小、代替案、停止を決める場を作る |
| 誰が判断するか | 現場、発注側、営業、開発側の責任を分ける |
| 追加費用の扱い | 調査で増えた作業を無償対応にしない |

アジャイルは、責任を曖昧にする方法ではない。不確実性を可視化し、一定期間ごとに判断するための進め方だと思う。

## 完成責任の認識がずれると大きな問題になる

システム開発では、「何をもって完成とするか」がずれると大きな問題になる。

発注側は、「業務で使える状態になること」を完成だと考える。受注側は、「合意した仕様の範囲を実装し、検収条件を満たすこと」を完成だと考える。営業側は、「提案時に説明した効果が出ること」を期待として伝えていることがある。現場のエンジニアは、「技術的に実現できる範囲」と「業務上必要な範囲」の差に挟まれる。

この認識がそろっていないと、次のようなことが起こる。

- 調査が必要な作業なのに、完成保証つきの開発として扱われる
- 仕様にない処理が、当然できるものとして期待される
- 既存システムや既存データの問題が、開発側の責任として扱われる
- 発注側の意思決定が遅れても、スケジュール遅延だけが問題になる
- 営業時の説明と現場で可能な実装範囲がずれる
- 途中で分かった制約を共有しても、後から「最初にできると言った」と言われる

特に、既存システムや商用ソフトに依存する開発では、外部要因が多い。API、ライセンス、データ品質、既存DB、既存業務フロー、利用者の暗黙知などが絡むため、開発側だけで制御できないことがある。

## 過去の大型開発紛争から見えること

日本でも、大型システム開発が頓挫し、発注側とベンダー側で訴訟になった事例は報道されている。

たとえばNHKと日本IBMの営業基幹システム開発をめぐる訴訟では、NHK側が2025年2月に約54億7000万円の支払いを求めて提訴したと報道されている。一方で、日本IBM側は、開始後に現行システムの解析を実施した上で移行方針やスケジュールを確定する契約だった、という趣旨の見解を示したと報道されている。

参考:

- [NHK、システム開発めぐり日本IBMを提訴 - ITmedia NEWS](https://www.itmedia.co.jp/news/articles/2502/04/news169.html)
- [日本IBM、NHKシステムの開発中断について見解 - ITmedia NEWS](https://www.itmedia.co.jp/news/articles/2502/07/news182.html)

また、文化シヤッターと日本IBMのシステム開発をめぐる訴訟では、最高裁が2025年1月に文化シヤッター側の上告を棄却し、日本IBMに約20億円の支払いを命じた判決が確定したと報道されている。

参考:

- [文化シヤッターの損害賠償請求訴訟、日本IBMの20億円支払い確定 - 日刊鉄鋼新聞](https://www.japanmetaldaily.com/articles/-/231149)

ここで個別事件の責任をこの記事で断定するつもりはない。重要なのは、大型開発では「何が契約範囲だったのか」「どこまでが調査だったのか」「既存システムの解析結果を踏まえてどう変更すべきだったのか」「完成責任を誰がどこまで負うのか」が大きな争点になり得るということだ。

製造業の3D自動化でも、規模は違っても同じ構造が起こり得る。

## 営業側にも理解してもらう必要がある

技術的に不確実性が高い案件では、営業側の説明も重要になる。

営業段階で「できます」と強く言い切ると、発注側は完成を前提に予算やスケジュールを組む。しかし現場では、実データ、API、ライセンス、既存システムを調べて初めて分かることがある。

そのため、営業側も次のように説明した方がよい。

- まず実現可能性を確認するフェーズが必要
- PoCの結果によって、実装範囲を調整する
- できること、できないこと、追加調査が必要なことを分ける
- 固定価格で完成保証できる範囲と、探索が必要な範囲を分ける
- 既存データや既存システムに起因する問題は、別途判断が必要
- 一定期間ごとに継続、縮小、代替案、停止を判断する

営業がこの前提を握らずに受注すると、現場のエンジニアが後から無理な約束を背負うことになる。

## 発注側にも必要な責任がある

発注側にも、開発を進める上で必要な責任がある。

もちろん、技術実装は開発側の責任だ。しかし、業務要件、既存データ、検収条件、優先順位、判断タイミングは発注側が関わらないと決められない。

発注側に必要なのは、たとえば次のようなことだ。

- 実データを提供する
- 正常データと異常データの例を出す
- 何をもって成功とするかを決める
- どの失敗なら許容できるかを決める
- 現場の暗黙知を共有する
- スプリントごとの判断に参加する
- 追加費用や範囲変更の判断を先送りしない

発注側が「全部お任せ」で、あとから「業務で使えない」と判断すると、開発側との認識差が大きくなる。探索型の開発では、発注側も一緒に判断する必要がある。

## 最初に決めておきたいこと

不確実性が高い開発では、契約や進め方の段階で次の点を明確にした方がよい。

| 項目 | 決めること |
| --- | --- |
| フェーズ分け | 調査、PoC、本開発、運用を分ける |
| 成果物 | 調査報告、検証コード、プロトタイプ、本番機能を区別する |
| 完了条件 | 何ができればそのフェーズを完了とするか |
| 除外条件 | 実現困難なものをどう扱うか |
| 判断会議 | 継続、縮小、停止、追加調査をいつ決めるか |
| 責任分界 | API制約、ライセンス制約、データ不備を誰が判断するか |
| 追加費用 | 想定外の調査や実装が出た場合の扱い |
| 検収条件 | 技術的成功と業務上の利用可能性をどう分けるか |

ここを曖昧にしたまま進めると、後から「完成していない」「仕様が違う」「追加費用は払えない」「最初から分かっていたはず」という話になりやすい。

## まとめ

不確実性が高い開発では、PoCとアジャイルな進め方が必要になる。

ただし、アジャイルは「何となく作りながら進める」という意味ではない。できること、できないこと、追加調査が必要なことを短いサイクルで確認し、範囲と優先順位を更新していく進め方だ。

特にCAD自動化や既存システム刷新のような領域では、外部ソフト、ライセンス、既存データ、既存業務、暗黙知が絡む。開発側だけで制御できない問題も多い。

だからこそ、発注側、受注側、営業側、現場のエンジニアが、探索型の開発であることを最初に共有する必要がある。固定仕様の完成責任として扱える範囲と、調査しながら判断する範囲を分けないと、大きなトラブルになりやすい。

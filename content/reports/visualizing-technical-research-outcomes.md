結論から言うと、技術調査の成果は「調べました」だけでは伝わりにくい。

技術調査の価値は、次に進めるか、追加確認が必要か、保留すべきか、別案へ切り替えるべきかを判断できる形に整理して初めて伝わりやすくなる。

この記事では、技術調査の成果を、関係者が判断しやすい形で可視化する方法を整理する。

PoC、撤退条件、外部依存、責任分界点、非機能要件などの用語は [開発・クラウド・CADの記事で使う用語メモ](/notes/system-development-terms) に短く整理している。

## なぜ技術調査の成果は見えにくいのか

実装した画面や動くデモは、成果として分かりやすい。

一方で、技術調査は成果が見えにくい。理由は、調査の価値が「作ったもの」ではなく「判断材料」にあるからだ。

たとえば、次のような調査結果は重要だが、成果として伝わりにくい。

- このAPIでは必要な情報を取得できない
- 無料枠では検証できるが、本番では費用が合わない
- ローカルでは動くが、本番ランタイムでは動かない
- サンプルデータでは成功するが、実データでは例外が多い
- 完全自動化は難しいが、人の確認を残せば使える可能性がある
- ライセンス上、その使い方は確認が必要

これらは「何も作れなかった」ではない。むしろ、後から大きな問題になる可能性を早めに見つけた成果だ。

ただし、その価値を伝えるには、調査結果を判断できる形に整理する必要がある。

## 「分かったこと」と「判断」を分ける

技術調査では、分かった事実と、それに基づく判断を分けて書いた方がよい。

たとえば、次のように整理できる。

| 項目 | 内容 |
| --- | --- |
| 分かったこと | エッジ系の実行環境では対象ライブラリの一部機能が動かない |
| 影響 | 本番環境をその実行環境にする場合、そのままでは採用できない |
| 判断 | Node.js Runtimeを使う案、または別ライブラリを検討する |
| 次に見ること | Node.js系の実行環境や別のホスティング方式で同じ処理が動くか確認する |

「動かなかった」だけで終わると失敗に見えやすい。

しかし、どの環境で、何が、なぜ問題になり、次に何を見るべきかまで整理すると、判断材料になる。

## 判断ステータスで整理する

技術調査の結果は、採用か不採用かの二択だけではない。

現実には、次のような段階で整理した方が分かりやすい。

| 判断 | 意味 |
| --- | --- |
| 採用候補 | 主要なリスクが許容範囲で、次の検証や本開発に進める |
| 条件付き採用 | 追加確認や制約対応が終われば進められる |
| 保留 | 判断材料が足りず、今は決めない |
| 撤退 | 現在の案では重要条件を満たせない |
| 別案検討 | 目的は残すが、方法を変える |

この分類を使うと、「できる / できない」よりも現実に近い説明ができる。

たとえば、AI分類の調査であれば、次のように書ける。

| 判断 | 内容 |
| --- | --- |
| 採用候補 | 90%以上の精度が出て、費用も許容範囲 |
| 条件付き採用 | 精度は出るが、機密情報の扱いを確認する必要がある |
| 保留 | データ件数が少なく、評価に必要なサンプルが足りない |
| 撤退 | 精度が80%未満で、業務利用には不足している |
| 別案検討 | AIは分類候補だけ出し、最終判断は人が行う |

このように整理すると、関係者は「次に何を決めればよいか」を見やすくなる。

## 観点ごとに評価する

技術調査では、技術的に動いたかだけを見ると判断を誤りやすい。

少なくとも、技術、費用、運用、データ、セキュリティ、責任範囲を分けて見たい。

| 観点 | 確認すること | 例 |
| --- | --- | --- |
| 技術 | 実現できるか | API、SDK、ライブラリ、実行環境 |
| 費用 | 継続できる金額か | 月額、従量課金、転送料、サポート費 |
| 運用 | 誰が維持できるか | 監視、ログ、障害対応、復旧手順 |
| データ | 本番データで使えるか | 件数、品質、権限、バックアップ |
| セキュリティ | 扱ってよい構成か | 機密情報、個人情報、シークレット管理 |
| 契約 | 利用条件に合うか | ライセンス、商用利用、外部送信 |
| 責任 | 誰が判断するか | 業務上の正しさ、障害時対応、受け入れ条件 |

この表を使うと、技術者だけでなく、発注側、営業側、運用側とも話しやすくなる。

## 調査結果を1枚で見える形にする

技術調査の報告では、細かいログや試行錯誤をすべて並べるより、最初に判断サマリーを置くと伝わりやすい。

たとえば、次のような形にする。

| 項目 | 内容 |
| --- | --- |
| 調査テーマ | PDF分類にAIを使えるか |
| 目的 | 確認作業の時間を減らす |
| 結論 | 完全自動化は保留。候補提示としては採用候補 |
| 成功したこと | 50件中43件を正しく分類できた |
| 問題になったこと | 誤分類時の検出方法、機密情報の扱い、月額費用 |
| 判断 | AIは候補提示に限定し、人が最終確認する案を検討 |
| 次に見ること | データ件数を増やした評価、利用規約、費用見積もり |

この形にすると、「何を調べたのか」「何が分かったのか」「次に何を判断するのか」が見えやすい。

詳細なログ、スクリーンショット、エラー内容、参考リンクは、その後に添付すればよい。

## 成功だけでなく未確認事項を書く

技術調査で重要なのは、成功したことだけではない。

未確認事項を明確にしておくことで、後から「そこも確認済みだと思っていた」というすれ違いを減らせる。

たとえば、次のように書く。

| 区分 | 内容 |
| --- | --- |
| 確認済み | 小規模データでは処理できた |
| 未確認 | 本番データ量、長時間運用、障害時復旧 |
| リスク | データ量が増えると費用が上がる可能性 |
| 追加確認 | 1万件データでの処理時間、月額費用、ログ確認 |

「確認済み」と「未確認」を分けるだけで、調査結果の信頼性は上がる。

## 採用しなかった案も残す

技術調査では、採用した案だけでなく、採用しなかった案も残した方がよい。

理由は、後から同じ案を再検討するときに、過去の判断理由が分かるからだ。

| 案 | 判断 | 理由 |
| --- | --- | --- |
| A案 | 保留 | 実行環境差分を追加確認する必要がある |
| B案 | 採用候補 | 要件との整合性が高く、本番確認もしやすい |
| C案 | 条件付き採用 | 追加コストや運用確認が必要 |
| D案 | 別案検討 | セキュリティ要件に応じて検討する |

採用しなかった案を残すことは、批判ではない。判断の前提を残す作業だ。

この考え方は、クラウド、DB、AI、CAD、外部APIのどれでも使える。

## 報告文に落とすときの形

会社やチームに報告する場合は、調査内容を長く書くより、判断に必要な情報を短くまとめたい。

たとえば、次のような形にする。

```text
【調査テーマ】
PDF分類にAIを利用できるか

【目的】
確認作業の時間を減らすため、AIによる分類の実現可能性を確認する

【確認したこと】
・50件のテストデータで分類精度を確認
・API料金と月額費用の概算を確認
・機密情報を外部APIへ送る場合の注意点を整理

【結果】
・50件中43件を正しく分類
・完全自動化には精度と責任範囲の課題あり
・候補提示として使う案は検討可能

【判断】
条件付き採用

【次に確認すること】
・テストデータを増やした精度確認
・利用規約と情報管理上の確認
・人による最終確認を残す運用案の整理
```

この形なら、「何をしたか」だけでなく「どう判断したか」が伝わりやすい。

## 技術調査の成果物として残したいもの

技術調査では、次のような成果物を残すと後から使いやすい。

- 調査サマリー
- 判断ステータス
- 成功条件
- 撤退条件
- 未確認事項
- 採用しなかった案
- 追加確認事項
- 参考リンク
- スクリーンショット
- 再現手順

すべてを毎回細かく作る必要はない。ただし、最低限「結論」「判断理由」「次に見ること」は残したい。

## まとめ

技術調査の成果は、動くデモやコードだけではない。

調査によって分かった制約、未確認事項、採用しなかった案、撤退条件、別案候補も重要な成果である。

技術調査を可視化するには、「採用候補」「条件付き採用」「保留」「撤退」「別案検討」のように、次の意思決定につながる形で整理するとよい。

「調べました」で終わらせず、「この条件なら進める」「ここは追加確認が必要」「この案は保留する」「別案へ切り替える」と言える状態にすることが、技術調査の価値を伝えるために重要だと考える。

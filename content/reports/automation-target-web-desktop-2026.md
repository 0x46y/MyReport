業務の自動化を考えるとき、最初に確認した方がよいのは「何を操作したいのか」だと思う。

同じ自動化でも、ブラウザで動くWebアプリを操作するのか、ユーザーのローカルPC上で動くデスクトップアプリを操作するのかで、難易度や安定性がかなり変わる。ここを先に整理しておくと、自動化の相談が来たときに「この種類なら比較的安定してできる」「この種類は調査が必要」「この方法だと壊れやすい」と説明しやすくなる。

## 操作対象を先に分ける

自動化の対象は、大きく次のように分けて考えられる。

| 対象 | 例 | 自動化のしやすさ | 主な手段 |
| --- | --- | --- | --- |
| Webアプリ | 社内Webシステム、管理画面、入力フォーム | 比較的やりやすい | Playwright、Selenium |
| Windowsデスクトップアプリ | 業務用クライアント、古い社内アプリ | アプリの作りによる | pywinauto、UI Automation、uiautomation |
| 画面上の画像や座標 | UI要素を取得できない画面 | 壊れやすい | pyautogui、画像認識、座標クリック |
| キーボード操作中心 | Tab、Enter、ショートカットで進める画面 | 条件付きで使える | sendkeys、ホットキー |

自動化は、上から順に安定しやすく、下に行くほど画面の見た目やタイミングに影響されやすい。最初から画像認識や座標クリックで作るより、まずは対象アプリが内部的にどのようなUI要素を持っているかを確認する方がよい。

## Webアプリは比較的自動化しやすい

ブラウザで動作するWebアプリは、自動化しやすいことが多い。

理由は、画面上のボタン、入力欄、リンク、テーブルなどがHTMLのDOM要素として存在しているからだ。ブラウザの開発者ツールを使えば、操作したいボタンや入力欄のセレクターを調べられる。id、name、class、placeholder、aria-label、表示テキストなどを使って、特定の要素を指定できる。

たとえば、ログイン画面でユーザー名を入力し、パスワードを入力し、ログインボタンを押す処理は、PlaywrightやSeleniumで比較的素直に書ける。人間が画面を見ながらクリックするのではなく、プログラムがHTML要素を直接探して操作するため、座標クリックより安定しやすい。

最近のClaude CodeやCodexのようなAIコーディングエージェントも、Webアプリの確認ではこの方向が強いと感じる。AIが自分で作ったWebアプリをPlaywrightで開き、ボタンを押したり、フォームを入力したり、エラーが出ていないか確認したりできる。Computer Use系の機能も、画面操作という意味では近い考え方だ。

Webアプリ自動化で使う代表的なツールは次のようなものだ。

| ツール | 特徴 |
| --- | --- |
| Playwright | モダンなWebアプリのテストや操作に向いている。待機処理やブラウザ制御が扱いやすい |
| Selenium | 長く使われているブラウザ自動化ツール。対応範囲が広く、情報も多い |
| ブラウザ開発者ツール | セレクター確認、通信確認、エラー確認に使う |

ただし、Webアプリだから必ず簡単というわけではない。ログイン認証、二要素認証、CAPTCHA、動的に変わる画面、遅延読み込み、権限によって変わるメニューなどがあると、考慮することは増える。

また、バックグラウンドで自動化を実行する場合は、エラー時の情報を残す仕組みが重要になる。画面を見ていれば「このメッセージが出て止まった」と分かるが、裏で実行していると原因が分かりにくい。失敗時にスクリーンショットを保存する、HTMLを保存する、ログを残す、どの手順で止まったかを出す、という仕組みを入れておくと調査しやすい。

## デスクトップアプリは調査が重要

ローカルPCで動くWindowsデスクトップアプリを自動化したい場合は、いくつかのパターンがある。

最初に確認したいのは、そのアプリの画面要素を外部から取得できるかどうかだ。ボタン、入力欄、リスト、メニューなどがUI AutomationやWin32の情報として見えるなら、比較的安定して操作できる可能性がある。

調査に使えるツールとしては、Microsoftが提供しているSpy++やInspect.exe、Accessibility Insights for Windowsなどがある。Spy++はウィンドウハンドルやメッセージ、クラス名などを確認するのに使える。UI Automationの情報を見たい場合は、Inspect.exeやAccessibility Insightsの方が向いている場面もある。

Pythonで操作する場合、自分はまずpywinautoを候補にする。

pywinautoはWindowsアプリの画面要素を探し、ボタンを押したり、入力欄に文字を入れたり、ウィンドウを操作したりできる。Win32系のアプリにも、UI Automation系のアプリにも対応しやすく、Pythonで業務自動化を書くときの入り口として使いやすい。

## UI Automationが使えるなら安定しやすい

デスクトップアプリで安定した自動化を作るなら、UI Automationの情報が取れるかどうかが大事になる。

UI Automationでボタン名、コントロールタイプ、AutomationId、階層構造などが見える場合、画像や座標に頼らずに操作できる。たとえば「OKボタンを押す」「一覧の3行目を選ぶ」「特定の名前のテキストボックスに入力する」といった操作を、画面位置ではなく要素情報で指定できる。

pywinautoで扱いづらい構造の場合、Pythonのuiautomationライブラリを使う選択肢もある。よりUI Automation寄りに細かく要素を探したいとき、リストやツリー、独自コントロールが絡むときに役立つ場合がある。

ただし、デスクトップアプリはWebアプリよりもアプリごとの差が大きい。古いアプリ、独自UI部品を使っているアプリ、描画だけで作られているアプリでは、外部からボタンやリストとして認識できないこともある。その場合は、UI Automationだけで完結しない。

## 最後の手段として画像認識やキー操作を使う

UI要素が取れない場合は、pyautoguiで画面のスクリーンショットを取り、特定の画像を探してクリックする方法がある。たとえば、保存ボタンの画像を用意しておき、画面上から一致する場所を探して押すような方法だ。

この方法は、人間の見た目に近い操作ができる一方で、壊れやすい。画面解像度、拡大率、テーマ、ウィンドウ位置、ボタンの色、処理タイミングが変わると失敗しやすい。リモートデスクトップ環境や複数モニター環境でも注意が必要になる。

キーボード操作も選択肢になる。Tabキーでフォーカスを移動し、Enterで決定し、ショートカットキーで画面を進める方法だ。画面要素が取れない場合でも動くことがある。ただし、途中でフォーカスがずれると別のボタンを押してしまう危険があるため、現在の画面状態を確認する処理とセットで使いたい。

## 自動化が失敗しやすい原因

自動化が失敗する原因は、だいたい決まっている。

- 画面の読み込みが終わる前に次の操作をしている
- ログイン状態や権限によって表示される画面が違う
- ボタン名やHTML構造が変更された
- ウィンドウが裏に回っている
- 予期しないダイアログが表示されている
- エラーメッセージが出ているのに処理が進もうとしている
- 画面解像度や拡大率が変わった
- ネットワークやサーバー応答が遅い
- 入力データの形式が想定と違う
- 手動操作が途中で入って状態が変わった

そのため、自動化では「正常に進む処理」だけでなく「失敗したときにどう分かるようにするか」が重要になる。スクリーンショット、ログ、実行ステップ名、入力データ、エラー時の画面タイトルなどを残しておくと、原因調査がかなり楽になる。

## Power Automate Desktopを使って感じたこと

Power Automate Desktopも、業務自動化の選択肢としては有力だ。画面操作を記録でき、GUIでフローを作れるため、プログラムを書かない人でも入りやすい。Microsoft製品との連携や、社内でRPAとして説明しやすい点もメリットだと思う。

一方で、自分が細かい制御をしたい場合は、PythonでpywinautoやPlaywrightを使う方が合っていると感じた。

理由は、コードとして管理しやすいからだ。条件分岐、例外処理、ログ出力、スクリーンショット保存、リトライ、データ加工、テスト実行、Git管理などを考えると、Pythonの方が自分には見通しがよい。特に、失敗したときにどこで止まったのかを細かく追いたい場合や、既存のPython処理と組み合わせたい場合は、コードで書ける方が扱いやすい。

ただし、Power Automate Desktopが悪いという話ではない。簡単な定型作業、Microsoft 365との連携、利用者自身が修正する前提の小さな自動化では便利だと思う。逆に、長く保守する処理や複雑な条件分岐がある処理では、Pythonで作る方が向いている場面がある。

## 自動化の相談を受けたときの確認項目

自動化の話が来た場合、自分なら最初に次の点を確認する。

| 確認項目 | 見る理由 |
| --- | --- |
| 操作対象はWebかデスクトップアプリか | 使う技術と難易度が変わる |
| 画面要素を取得できるか | DOM、UI Automation、Win32情報が取れるか確認する |
| ログインや認証があるか | 二要素認証やCAPTCHAがあると自動化しにくい |
| 処理件数と実行頻度 | 1日1回なのか、大量処理なのかで設計が変わる |
| 失敗時に止めるか続けるか | エラーハンドリングの方針が変わる |
| 人の確認が必要か | 完全自動か、途中承認ありかを決める |
| 実行環境はどこか | ローカルPC、サーバー、仮想環境で制約が違う |
| 保守する人は誰か | GUIツールかコード管理かの判断に関わる |

この確認を先にしておくと、「とりあえず自動化できます」と曖昧に答えるのではなく、「WebアプリなのでPlaywrightでいけそうです」「デスクトップアプリなのでUI Automationで要素が取れるか調査します」「画像認識になる場合は壊れやすいので注意が必要です」と説明できる。

## まとめ

自動化は、対象によってできることと難しいことが大きく変わる。

WebアプリはDOM要素を指定できるため、PlaywrightやSeleniumで比較的安定して操作しやすい。デスクトップアプリは、UI AutomationやWin32情報が取れるかどうかが重要になる。pywinautoやuiautomationで要素を扱えるなら安定しやすいが、要素が取れない場合は画像認識やキー操作に寄せる必要があり、壊れやすくなる。

自動化を作るときは、操作できるかどうかだけでなく、失敗したときに原因が分かるか、保守できるか、実行環境が安定しているかも見る必要がある。自分としては、まず対象を分類し、使える情報を調査し、なるべくDOMやUI Automationのような構造化された情報で操作する方針を取るのがよいと考えている。

クラウドは初期導入を軽くできる一方で、長期的なランニングコストと依存関係を見落とすと、後から選択肢が狭くなる。

技術的に動くことと、5年運用できることは別問題だ。クラウドを使う価値は大きいが、月額費用、転送量、ログ、監視、バックアップ、権限管理、担当者変更、移行コストまで含めて見ないと、実際の負担を判断しにくい。

## なぜクラウドコストは見落とされやすいのか

クラウドのコストは、開発初期には小さく見えやすい。

PoCや個人検証の段階では、利用者も少なく、保存データも少なく、アクセス量も限られている。そのため、最初は無料枠や数百円の範囲で収まることが多い。しかし、本番運用になると、利用者、データ量、ログ、監視、バックアップ、転送量、API呼び出し回数が増える。

見落としやすい理由は複数ある。

- 開発担当者と運用担当者が違う
- 作った人が後から別案件へ移動する
- PoC時点では利用量が少ない
- 無料枠の範囲だけを見て判断してしまう
- 料金表がサービスごとに分かれていて全体像を見にくい
- 開発中に詰まったとき、課金すれば一時的に進められることがある
- クラウド事業者の説明は、自社サービスの強みを中心に見えやすい

これはクラウド事業者が悪いという話ではない。クラウドやSaaSは、自前で用意すると大変な機能を短時間で使えるようにしてくれる。その価値は大きい。ただし、利用する側は、自分たちの利用量と運用体制で費用を見積もる必要がある。

## 無料で使えることと、実際に使えることは違う

無料枠は技術検証では重要だが、無料枠があることと、検証が止まらずに進むことは同じではない。

たとえば、無料対象のリソースが用意されていても、利用したいリージョンやSKUで空きがなかったり、別の設定で課金が発生したり、無料枠の条件を読み違えたりすることがある。

自分がAzureを少し試した範囲では、無料対象のVM系リソースを作成しようとしたときに、利用可能なリージョンやSKUが見つからず、検証を進めにくいことがあった。これはAzure全体が使いにくいという意味ではない。既存のMicrosoft系業務基盤と組み合わせるならAzureを選ぶ理由は強い。ただ、個人検証では「無料枠がある」だけでなく「今すぐ使えるか」も判断材料になる。

場合によっては、無料枠を探して時間を使うより、数百円から数千円を払って確実に検証できる環境を使う方が安いこともある。技術調査では、お金だけでなく、調査に使う時間もコストとして見る必要がある。

無料枠を使って検証するときの具体的な確認項目は [無料枠で検証するときに確認したいこと](/reports/free-tier-verification-checklist) に分けて整理している。

## コストを見るときの観点

クラウド費用は、月額のサーバー代だけではない。

| 観点 | 確認すること |
| --- | --- |
| Compute | VM、コンテナ、サーバーレス、ジョブ実行時間 |
| Storage | 保存容量、最低保存期間、バックアップ、スナップショット |
| Network | 外向き転送、リージョン間転送、CDN、NAT |
| Database | 読み書き回数、ストレージ、バックアップ、レプリカ |
| Logs | ログ保存量、保持期間、検索、監視 |
| AI/API | 呼び出し回数、トークン量、レート制限 |
| Security | 権限管理、監査ログ、WAF、Zero Trust |
| Operations | 担当者、障害対応、問い合わせ、ドキュメント |

特にWeb配信では、保存容量より外向き転送が効くことがある。AI機能では、1回の処理は安く見えても、利用者数や再試行回数が増えると費用が読みにくくなる。

## ベンダーロックインは悪ではないが、前提に入れる

クラウドやSaaSを使うことは、自前で実装する範囲を減らすことでもある。認証、データベース、ファイル配信、監視、AI API、ノーコード基盤などを外部サービスに任せれば、開発は速くなる。

一方で、便利な機能を使うほど、そのサービスの仕様、料金、制限、ロードマップに依存する。

確認したいこと:

- データを外へ取り出せるか
- 別サービスへ移行できる形式か
- API制限やレート制限はあるか
- 料金が上がったときに代替案があるか
- そのサービスを使える担当者が複数いるか
- 契約やライセンス上の制約は問題ないか
- サポート終了や仕様変更が起きたときに対応できるか

ベンダーロックインは必ず避けるべきものではない。商用サービスを使うことで、運用負荷を下げ、短期間で価値を出せる場合も多い。重要なのは、依存していることを認識し、どこまで許容するかを決めることだ。

## 技術選定で確認したいランニングコスト

クラウドを採用する前に、少なくとも次の3つの期間で見積もりたい。

| 期間 | 見ること |
| --- | --- |
| 初期検証 | 無料枠で試せるか、少額課金で確実に試す方がよいか |
| 本番初期 | 月額費用、転送量、ログ、バックアップ、監視 |
| 3年後 | データ量、利用者数、担当者変更、移行可能性、契約変更 |

特に企業システムでは、初期費用だけでなく、3年から5年の運用費用を見た方がよい。最初は安く見えても、データ量や転送量が増えたあとに移行しようとすると、移行コストが大きくなる。

## 判断材料として残したいこと

クラウド選定では、次のように記録しておくと後から説明しやすい。

- 候補にしたサービス
- 採用したサービス
- 採用した理由
- 採用しなかった理由
- 月額費用の見積もり
- 無料枠で試せる範囲
- 無料枠を超えたときの料金
- 将来移行するときの難しさ
- 運用担当者と障害時の対応
- 撤退条件

「Cloudflareが安い」「AWSが高い」のような単純な比較ではなく、自分たちの使い方でどうなるかを見たい。既存システムがAWSやAzureで組まれているなら、Cloudflareの方が軽いからという理由だけで置き換えるのは危ない。移行コスト、既存の監査、権限管理、担当者の学習コストまで含める必要がある。

## まとめ

クラウドは、初期導入を軽くし、短時間で多くの機能を使えるようにしてくれる。その価値は大きい。

ただし、便利な機能を使うほど、月額費用、外向き転送、ログ、監視、バックアップ、API利用量、ベンダー依存が積み上がる。技術的に動いた時点で判断を止めるのではなく、運用できるか、費用を説明できるか、将来移行できるかまで見た方がよい。

クラウド選定で大事なのは、最安のサービスを探すことではない。今の要件、運用体制、既存システム、将来の変更可能性に対して、どこまで依存してよいかを決めることだ。

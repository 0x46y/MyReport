結論から言うと、システム開発を依頼するときは「何を作りたいか」だけでは足りない。

誰が使うのか、誰が情報を更新するのか、データを誰が管理するのか、完成後に誰が保守するのか、URLやドメインを誰が管理するのか、どのような依頼として相談するのかを先に整理しておくと、発注側と開発側の認識を合わせやすくなる。

この記事では、システム開発の発注経験が少ない人に向けて、最初に決めておきたいことを整理する。

ドメイン、CSV、JSON、拡張子、請負、準委任、保守などの用語は [開発・クラウド・CADの記事で使う用語メモ](/notes/system-development-terms) に短く整理している。

## なぜ「何を作るか」だけでは足りないのか

システム開発では、「予約サイトを作りたい」「地図で施設を見られるようにしたい」「申請をWeb化したい」のように、作りたいものから話が始まることが多い。

もちろん、作りたいものを説明することは重要だ。

ただし、それだけでは開発に必要な情報として足りない場合がある。

確認したいこと:

- 誰が使うのか
- 何の作業を減らしたいのか
- 誰が情報を更新するのか
- データはどこから来るのか
- 間違った情報が出たとき誰が直すのか
- 公開後の問い合わせは誰が受けるのか
- どこまでできれば完成なのか
- 完成後の保守費用を誰が持つのか

作る前にこれらを整理しておくと、後から「そこまで含まれていると思っていた」という認識違いを減らしやすい。

## 最初に整理したいこと

開発会社や個人開発者へ相談する前に、次のような項目を整理しておくと話が進めやすい。

| 項目 | 確認したいこと |
| --- | --- |
| 目的 | 何を解決したいのか、何の作業を減らしたいのか |
| 利用者 | 誰が使うのか、一般公開か、社内利用か |
| 更新者 | 誰が情報を追加・修正するのか |
| データ管理者 | データの正しさを誰が確認するのか |
| 保守担当 | 公開後の不具合、問い合わせ、更新を誰が見るのか |
| 予算 | 初期開発費だけでなく、月額費用や保守費用をどう見るか |
| 納期 | いつまでに何が使える状態ならよいか |
| URL・ドメイン | 独自ドメインを取るか、サービス側のURLを使うか、誰が管理するか |
| 依頼の仕方 | 完成物を依頼するのか、調査や短期検証から始めるのか |

この表は、最初から完璧に埋める必要はない。分からないところを明確にすることにも意味がある。

## 利用者と更新者を分けて考える

Webシステムでは、見る人と更新する人が違うことが多い。

たとえば、一般利用者は画面を見るだけだが、施設の担当者は情報を更新する。管理者は全体を確認する。開発者は不具合や機能追加を対応する。

この役割を分けておかないと、次のような問題が起きやすい。

- 誰でも情報を編集できてしまう
- 更新担当者が分からず、古い情報が残る
- 利用者からの問い合わせ先が決まっていない
- 管理画面が必要かどうか後から分かる
- 権限管理が必要になり、追加開発が発生する

最初に「見る人」「更新する人」「管理する人」を分けておくと、画面、ログイン、権限、運用の設計を考えやすくなる。

## データを誰が管理するか

システムは、画面だけでなくデータが重要になる。

施設一覧、商品一覧、予約情報、問い合わせ内容、ユーザー情報など、どのデータを誰が管理するのかを決めておきたい。

確認したいこと:

- 最初のデータは誰が用意するか
- Excel、CSV、JSON、手入力のどれで管理するか
- データが間違っていた場合、誰が直すか
- 更新頻度はどれくらいか
- 個人情報や機密情報を含むか
- バックアップは必要か
- データを将来別のシステムへ移せるか

Excelで管理している情報をWebシステムに入れる場合でも、そのまま使えるとは限らない。表の列名、入力ゆれ、空欄、重複、古い情報があると、取り込み前に整理が必要になる。

CSVやJSONのような形式を使う場合も、発注側にとっては聞き慣れないことがある。その場合は、開発側が「Excelの表をシンプルに保存した形式」「プログラムが読みやすいデータ形式」のように説明しながら進めると認識を合わせやすい。

## URLやドメインを誰が管理するか

小規模なWebシステムでは、最初はサービス側が用意したURLで公開できる場合がある。

たとえば、`example.workers.dev` や `example.vercel.app` のようなURLでも公開自体はできる。

一方で、長く使うサイトなら独自ドメインを取得する価値がある。

独自ドメインを使う理由:

- 利用者に案内しやすい
- サービスを引っ越しても同じURLを使いやすい
- 名刺、チラシ、メール、SNSに載せやすい
- 組織やサービスとして信頼感を出しやすい
- メールアドレスやサブドメインにも展開しやすい

ドメインは、Webサイトの住所のようなものだ。

裏側のサーバーやホスティング先を変えても、ドメインを持っていれば、利用者に案内するURLを変えずに済む場合がある。これは、引っ越しても同じ電話番号や窓口名を使えるようにしておく考え方に近い。

ただし、ドメインにも年額費用、更新管理、DNS設定、所有者管理がある。誰の名義で取得し、誰が更新するのかは先に決めたい。

## 完成物を依頼するのか、調査を依頼するのか

システム開発を外部へ依頼する場合、何を依頼するのかを先に確認したい。

大きく見ると、完成した成果物に対して対価を払う考え方と、一定期間の専門的な作業や支援に対して対価を払う考え方がある。

たとえば、「このWebサイトを作ってほしい」と「この方法で実現できるか調査してほしい」では、依頼の性質が違う。

一般には、完成した成果物に対して対価を払う考え方に近いものとして請負、専門的な作業や支援に対して対価を払う考え方に近いものとして準委任という言葉が使われることがある。

ただし、契約の細かい判断は法律や契約実務に関わる。この記事では法的な結論は扱わず、発注前に確認したい観点として整理する。

確認したいこと:

- 完成物を決めて依頼するのか
- 調査や試行錯誤を含む依頼なのか
- 完成責任をどこまで求めるのか
- 仕様変更や追加要望が出た場合どう扱うか
- 1か月単位、1週間単位、短期検証などの進め方は可能か
- 検収条件をどう決めるか
- 保守や運用は別契約にするか

要件が明確で、作るものが固定されている場合は、完成物を基準にした進め方が合うことがある。

一方で、何が実現できるか分からない、実データを見ないと判断できない、利用者の反応を見ながら変えたい、という場合は、最初から完成物を固定しすぎると認識がずれやすい。

この点は、[不確実性が高い開発で、先に決めておきたいこと](/reports/exploratory-development-contract-risk-agile) や [小さな検証（PoC）が成功しても本番導入できない理由](/reports/poc-success-production-adoption-risks) とも関係する。

## 保守と運用を決めておく

システムは作って終わりではない。

公開後には、情報更新、問い合わせ対応、不具合対応、サーバーやクラウド費用、ドメイン更新、セキュリティ更新、バックアップなどが必要になる。

確認したいこと:

- 誰が日々の情報を更新するか
- 不具合が出たとき誰に連絡するか
- 月額費用は誰が払うか
- ドメイン更新を誰が管理するか
- 管理画面のパスワードを誰が持つか
- 担当者が変わったとき引き継げるか
- データのバックアップをどうするか

特に小規模なシステムでは、最初は作った人が全部見ていることが多い。しかし長く使うなら、担当者が変わっても分かる資料や管理方法を残しておきたい。

## 開発会社や開発者へ相談するときに伝えたいこと

開発を相談するときは、最初から詳細な仕様書を作れなくてもよい。

ただし、次の情報があると、開発側は見積もりや進め方を考えやすくなる。

伝えたいこと:

- 解決したい課題
- 現在のやり方
- 困っている作業
- 利用者の種類
- 更新する人
- 使っているExcelや資料
- 公開範囲
- 予算感
- 希望時期
- 完成後の運用担当
- 絶対に必要な機能
- あれば嬉しい機能

特に大事なのは、「絶対に必要な機能」と「あれば嬉しい機能」を分けることだ。

全部を最初から作ろうとすると、費用も期間も大きくなりやすい。まずは小さく公開し、利用状況を見ながら改善する方が合う場合もある。

## 最初から全部決めなくてもよい

システム開発では、最初からすべての要件を確定できるとは限らない。

もちろん、目的、利用者、予算、公開範囲、保守担当のような基本的な前提は整理した方がよい。一方で、管理画面、権限管理、承認フロー、高度な検索、通知機能などを最初から全部作ると、費用も期間も大きくなりやすい。

まずは小規模に公開し、次のようなことを確認しながら改善していく方法もある。

- 本当に使われるか
- 情報更新を続けられるか
- 問い合わせや修正依頼がどれくらい来るか
- 保守負荷はどれくらいか
- 管理画面や権限管理が本当に必要か
- どの機能が優先度高く使われるか

特に小規模なWebアプリでは、最初から高度な機能を作るより、まず運用できるかを確認した方がよい場合がある。

これは「何も決めずに進める」という意味ではない。最初に決めるべきことと、使われ方を見ながら決めることを分けるということだ。

## よくある認識違い

発注側と開発側では、同じ言葉でも受け取り方が違うことがある。

| 言葉 | 認識違いが起きやすい点 |
| --- | --- |
| 簡単な画面 | 裏側にログイン、権限、データベース、更新機能が必要な場合がある |
| Excelを取り込む | 入力ゆれ、空欄、重複、形式変換が必要な場合がある |
| 地図に表示する | 住所、緯度経度、更新頻度、検索条件を考える必要がある |
| AIで自動化する | 正解判定、人による確認、誤り時の責任を決める必要がある |
| 公開するだけ | ドメイン、HTTPS、問い合わせ、保守、費用が関係する |
| 後で直せる | 変更しやすい設計と、変更費用の合意が必要になる |

こうした認識違いは、どちらかが悪いという話ではない。専門分野が違うため、最初に言葉をそろえる必要がある。

## 相談するときに持っていくメモ

ここまでの内容を、実際に相談するときのメモにすると次のようになる。

最初からすべてを埋める必要はない。分からない項目や未定の項目も、そのまま相談材料になる。

| 項目 | メモ |
| --- | --- |
| 作りたいもの |  |
| 解決したい課題 |  |
| 使う人 |  |
| 更新する人 |  |
| 管理するデータ |  |
| 今使っている資料 |  |
| 公開範囲 |  |
| ドメインの希望 |  |
| 希望時期 |  |
| 予算感 |  |
| 公開後の保守担当 |  |
| 絶対に必要な機能 |  |
| あれば嬉しい機能 |  |
| 未定・相談したいこと |  |

この表は、発注側だけで完成させる必要はない。

むしろ、未定の項目を開発側と一緒に整理するために使うとよい。

## まとめ

システム開発を依頼するときは、「何を作るか」だけでなく、「誰が使うか」「誰が更新するか」「誰がデータを管理するか」「誰が保守するか」を最初に整理したい。

ドメイン、データ形式、契約形態、完成後の運用も、後から問題になりやすい。

発注側がすべてを理解している必要はない。しかし、分からないことを分からないまま進めるのではなく、最初に確認項目として並べておくと、開発側も説明しやすくなる。

開発をスムーズに進めるには、発注側と開発側が同じ前提を見ながら、目的、範囲、費用、運用、保守を一緒に整理していくことが重要だと考える。

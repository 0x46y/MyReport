3Dデータを扱う自動化では、ファイル形式の違いがそのままできることの違いになる。

同じ「3Dファイル」でも、CATPart、CATProduct、STL、IGES、STEPでは持っている情報が違う。見た目として同じ形状に見えても、内部に残っている設計情報、履歴、曲面、ソリッド性、アセンブリ構造、属性情報は大きく違う。

この記事では、製造業の3D自動化でよく出てくる形式を、自動化や変換の観点で整理する。

## 形式ごとに役割が違う

| 形式 | 主な位置づけ | 特徴 |
| --- | --- | --- |
| CATPart | CATIA V5の部品ファイル | 部品単位のネイティブデータ。形状や設計情報を保持する |
| CATProduct | CATIA V5のアセンブリファイル | 複数のCATPartやサブアセンブリの構成を持つ |
| STL | 表面メッシュ | 三角形メッシュで表面形状を表す。設計履歴や厳密な曲面情報は基本的に持たない |
| IGES | CAD交換形式 | 古くからあるCAD交換形式。面、曲線、ワイヤーなどの交換に使われる |
| STEP | CAD交換形式 | ISO 10303として標準化された製品データ交換形式。CAD間のデータ交換で広く使われる |

CATPartやCATProductは、CATIA側のネイティブ形式として扱われる。外部から直接中身を自由に読むための汎用テキスト形式ではなく、基本的にはCATIAや対応する商用コンバーター、API、ビューアーを通して扱う前提になる。

一方で、STL、IGES、STEPは、CAD間の交換や表示、3Dプリント、解析、外部ツール連携のために使われることが多い。

## STLは軽く扱いやすいが、情報がかなり落ちる

STLは、3D形状を三角形の集合として表す形式だ。Library of Congressの形式説明でも、STLは3次元オブジェクトの表面形状を表すが、色、テクスチャ、その他の一般的なモデル属性は表現しない形式として説明されている。

参考:

- [STL File Format Family - Library of Congress](https://www.loc.gov/preservation/digital/formats/fdd/fdd000504.shtml)

STLは扱いやすい。多くのソフトで開けるし、3Dプリントや簡易表示では便利だ。ただし、製造業の設計データとして見ると、かなり情報が落ちる。

- 曲面は三角形メッシュに近似される
- フィーチャー履歴は残らない
- 寸法拘束やパラメータは残らない
- 部品構成やアセンブリ構造は基本的に持たない
- 穴、隙間、反転面、非多様体などがあると後工程で問題になる

そのため、STLを受け取って「元のCADと同じように編集したい」と考えると難しくなる。STLは表示や3Dプリントには向いていても、設計意図を保持した交換形式ではない。

## IGESは古くからある交換形式だが、万能ではない

IGESは、CAD間で形状を交換するために長く使われてきた形式だ。IGESはASCIIテキスト形式として説明されることが多く、曲線、サーフェス、ワイヤーフレーム、ソリッド表現などを交換できる。

参考:

- [IGES File Format](https://docs.fileformat.com/cad/iges/)

ただし、IGESで渡せば必ず元のCADと同じ状態になるわけではない。特に、パラメトリックな履歴、設計意図、部品間の関係、アセンブリ構造などは期待通り残らないことがある。

実務では、IGESで読み込んだ後に面が分断されていたり、ソリッドではなくサーフェス集合として扱われたり、隙間や自由エッジが発生したりすることがある。見た目は近くても、後工程でソリッド化や修復が必要になる場合がある。

## STEPは強力だが、出せるかどうかは環境に依存する

STEPは、ISO 10303として標準化された製品データ交換形式だ。NISTもSTEPを、製品モデルデータ交換のための基盤的な標準として説明している。

参考:

- [STEP at NIST](https://www.nist.gov/ctl/smart-connected-systems-division/smart-connected-manufacturing-systems-group/step-nist)
- [Data Exchange of Parametric CAD Models Using ISO 10303-108 - NIST](https://www.nist.gov/publications/data-exchange-parametric-cad-models-using-iso-10303-108)

STEPはCAD間の交換でよく使われる。IGESよりも製品データ交換として広い範囲を扱いやすく、B-Rep形状や属性情報、構成情報を扱える場合がある。

ただし、ここで注意が必要なのは「STEPなら何でも完全に残る」というわけではないことだ。

NISTの資料でも、CADシステムが生成する特徴ベースの製品形状モデルには、パラメータ、制約、フィーチャー、設計意図が含まれるが、従来の交換標準では純粋な形状情報の交換に寄りがちだったと説明されている。つまり、STEPでもどのAPを使うか、どのCADがどこまで対応しているか、どのライセンスでどの出力が可能かによって結果が変わる。

## ネイティブ形式と交換形式を混同しない

CATPartやCATProductのようなネイティブ形式は、そのCADが最も多くの情報を保持しやすい形式だ。一方で、STL、IGES、STEPのような交換形式は、他のソフトへ渡すための形式だ。

この違いを混同すると、自動化の見積もりや設計を誤りやすい。

| 観点 | ネイティブ形式 | 交換形式 |
| --- | --- | --- |
| 情報量 | 多いことが多い | 形式や出力設定によって落ちる |
| 互換性 | 元ソフトに強く依存する | 他ソフトで扱いやすい |
| 自動解析 | 専用APIやライブラリが必要になりやすい | 形式によっては外部ツールで扱いやすい |
| 設計履歴 | 残りやすい | 落ちやすい |
| ベンダーロックイン | 強くなりやすい | 弱められるが完全ではない |

自動化で重要なのは、「どの形式を受け取れるか」だけではなく、「その形式に必要な情報が残っているか」だ。

## 過去のOCCT/WASM検証で試したかったこと

過去にOCCTとWebAssemblyで試していたのは、まさにこの形式変換や品質劣化の問題だった。

CATIAなどから出力されたSTLやIGESを、OCCTとWebAssemblyでブラウザ上に読み込み、自由エッジ、穴、面の崩れ、ソリッド化の可否を診断しようとしていた。STLとして出力された時点で設計履歴は失われるが、それでも形状としてどこまで修復できるのか、どこで破綻するのかを確認する意味があった。

この検証から分かるのは、3Dデータ変換は単なるファイル変換ではないということだ。

同じ形に見えるデータでも、後工程で使えるデータかどうかは別問題になる。特に製造業では、変換後にソリッドとして閉じているか、体積が異常でないか、面やエッジが破綻していないかを確認する必要がある。

## まとめ

3Dファイル形式は、拡張子が違うだけではなく、持っている情報の種類が違う。

STLは扱いやすいが、表面メッシュであり、設計履歴や厳密な曲面情報は落ちる。IGESは古くからある交換形式だが、読み込み後に面やソリッドの修復が必要になることがある。STEPは強力な標準だが、CAD側の対応範囲、出力設定、ライセンスによって扱える情報が変わる。

製造業の3D自動化では、「どの形式で出せますか」だけでは足りない。その形式に必要な情報が残っているか、後工程で使える品質か、失敗時にどう検出するかまで考える必要がある。

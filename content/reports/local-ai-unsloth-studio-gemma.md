ローカル環境でAIを動かし、できるだけ無料で使える環境を作れないか検討した。今回はその第一歩として、Unsloth Studio を使って Google の Gemma 系モデルをローカルで動かす検証を行った。

目的は、クラウドAPIに毎回依存せず、自分のPC上でLLMを起動し、ブラウザから対話できる状態を作ることだった。特に、環境構築のしやすさ、ディスク容量の扱いやすさ、モデルの導入しやすさを確認した。

## 実施内容

今回の検証では、Windows 本体の Python 環境を直接汚さないように、WSL 上で独立した Python 環境を作った。Python パッケージ管理には `uv` を使い、必要な依存関係を分離して管理する方針にした。

| 項目 | 内容 |
| --- | --- |
| 実行環境 | WSL |
| Python 管理 | uv |
| Python バージョン | Python 3.13 |
| UI | Unsloth Studio |
| 目的 | ローカルLLMの起動とブラウザ操作の確認 |
| 試したモデル | Gemma 系モデル |

Unsloth Studio を採用した理由は、ローカルLLM環境を比較的扱いやすく構築できそうだったことと、ディスク容量を効率的に管理しやすいと考えたためである。

## モデル導入

最初は大きめの Gemma 系モデルを試した。当初は 16GB 程度のモデルをダウンロードして動作確認しようとしたが、タイムアウトにより失敗した。

そこで、まずは軽量版のモデルに切り替えた。3GB 程度のモデルをダウンロードし、ローカル環境で LLM が正常に起動することを確認した。

| 試行 | 結果 |
| --- | --- |
| 16GB 程度のモデル | ダウンロードまたは起動時にタイムアウト |
| 3GB 程度の軽量モデル | ダウンロードと起動に成功 |

この時点で、Unsloth Studio のブラウザUIからローカルLLMを操作できることを確認できた。

## 発生したトラブル

環境構築中、Unsloth Studio の UI 上で入力が受け付けられない問題が発生した。画面上の入力欄に文字を入れようとしても、正常にDOMへ接続できていないような挙動になり、チャット入力ができなかった。

最初は Unsloth Studio 側の不具合か、WSL 側の起動設定の問題かを疑った。しかし調査していくと、ブラウザの特定の拡張機能が干渉していることが分かった。

対応として、該当するブラウザ拡張機能を無効化したところ、接続エラー自体は解消した。

## 日本語入力の課題

接続エラーは解消したものの、日本語IMEとの相性問題は残った。入力フォームへ直接日本語を入力しようとすると、期待通りに入力できない。

暫定対応として、別のエディタで日本語の文章を作成し、それを Unsloth Studio の入力欄へ貼り付ける運用にした。この方法であれば、日本語のプロンプトを使ったチャット入力は可能だった。

また、音声入力機能を使うことで代替できる可能性も確認した。

| 課題 | 対応 |
| --- | --- |
| UI入力欄が反応しない | ブラウザ拡張機能を無効化 |
| 日本語IMEで直接入力できない | 別エディタで作成して貼り付け |
| 入力の代替手段 | 音声入力も利用可能 |

## 検証結果

今回の検証では、Unsloth Studio を使ってローカルLLMを起動し、軽量な Gemma 系モデルで動作確認するところまでは成功した。

一方で、UI入力まわりには注意が必要だった。特にブラウザ拡張機能との干渉、日本語IMEとの相性問題は、実際に使ってみないと気づきにくいポイントだった。

## 所感

Unsloth Studio は、ローカルでLLMを試す入口としてはかなり便利だと感じた。WSL と `uv` を組み合わせることで、Windows 側の環境を汚さずに検証できる点もよかった。

ただし、快適に使うにはモデルサイズ、GPUメモリ、ブラウザ環境、日本語入力の扱いを考える必要がある。特に日本語で日常的に使う場合、入力欄のIME相性は無視できない。

今回の結論としては、Unsloth Studio はローカルLLMの起動検証には有効だが、日本語入力やUIまわりの安定性を含めて、別の実行環境も比較した方がよいと感じた。

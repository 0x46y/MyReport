結論から言うと、NoSQLとRDBはどちらが優れているかではなく、扱うデータの性質と業務上の正しさで選ぶものだ。

SNS、チャット、画像投稿、ログのように、データ構造が柔軟で、素早くプロトタイプを作りたい用途ではNoSQLが向いている場合がある。一方で、注文、在庫、会計、契約、権限管理のように、整合性、トランザクション、ロールバック、複雑な検索が重要な用途ではRDBを候補に戻した方がよい。

NoSQL、RDB、トランザクション、ロールバック、JOIN などの用語は [開発・クラウド・CADの記事で使う用語メモ](/notes/system-development-terms) に短く整理している。

個人開発でSQLite、Prisma、Supabase、Firebaseなどをどう使い分けるかは [個人開発でDBをどう選ぶか: SQLite / Prisma / Supabase / Firebase](/reports/personal-development-database-selection) に分けて整理している。

## まず見るべき条件

データベースを選ぶときは、最初に次の条件を見る。

| 観点 | NoSQLが合いやすい場合 | RDBが合いやすい場合 |
| --- | --- | --- |
| データ構造 | ユーザーごとに持つ項目が変わる | 表の構造や関係が明確 |
| 開発速度 | プロトタイプを早く作りたい | 業務ルールを正確に表現したい |
| 整合性 | 多少の遅延や重複を許容できる | 正確な残高、在庫、履歴が必要 |
| 検索 | キーや単純な条件で読むことが多い | 複数テーブルをJOINして検索する |
| 更新 | 独立したドキュメントを更新する | 複数データをまとめて確定する |
| 変更頻度 | 仕様変更に合わせて柔軟に持ちたい | 型や制約でデータ品質を守りたい |

ここで重要なのは、初期開発の速さだけで決めないことだ。最初はNoSQLで素早く作れても、後から複雑な集計、権限、監査、整合性が必要になると、設計変更のコストが大きくなる。

## NoSQLが向いている用途

FirestoreのようなNoSQLドキュメントデータベースは、ユーザーごとのデータやイベントを柔軟に保存しやすい。

たとえば次のような用途では候補になりやすい。

- SNSの投稿やコメント
- チャットのメッセージ
- 画像投稿アプリ
- 通知
- アプリの設定
- ユーザーごとの簡単な状態管理
- ログやイベントデータ

Firebaseを使う場合、Authentication、Firestore、Storage、Hostingなどを組み合わせることで、ログイン、データ保存、画像保存、公開までを短い時間で試しやすい。Googleアカウントを持っている利用者も多いため、Googleログインを使うアプリでは入口を作りやすい。

ただし、NoSQLは自由に書ける分、データ構造の責任がアプリ側に寄りやすい。どのコレクションに何を置くのか、読み取り回数が増えないか、権限ルールをどう書くか、後から集計できるかを先に見たい。

## RDBが向いている用途

PostgreSQLのようなRDBは、表同士の関係、制約、トランザクションを使って、データの正しさを守りやすい。

たとえば次のような用途ではRDBを候補にしやすい。

- 受注管理
- 在庫管理
- 会計
- 請求
- 契約管理
- 権限管理
- 顧客管理
- 業務ワークフロー

これらの用途では、「途中まで更新されたが一部だけ失敗した」という状態が問題になりやすい。注文と在庫と請求をまとめて扱う場合、全部成功するか、失敗したら元に戻すかを明確にしたい。

RDBでは、テーブル設計、外部キー、ユニーク制約、トランザクション、JOINを使って、業務上の正しさを表現しやすい。SupabaseのようにPostgreSQLを前提にしたサービスを使えば、Webアプリ向けの認証やAPIと組み合わせながら、RDBの強みも使える。

## 用途ごとの選び方

ざっくり整理すると、次のようになる。

| 用途 | 第一候補 | 理由 |
| --- | --- | --- |
| SNSの投稿 | NoSQL | 投稿単位で読み書きしやすく、構造変更にも対応しやすい |
| チャット | NoSQL | メッセージを時系列に追加しやすい |
| 画像投稿アプリ | NoSQL + オブジェクトストレージ | メタデータとファイルを分けて扱いやすい |
| アクセスログ | NoSQL / 分析基盤 | 書き込み量が多く、厳密なJOINより蓄積が重要 |
| ECの注文 | RDB | 注文、商品、在庫、支払いの整合性が重要 |
| 在庫管理 | RDB | 数量の増減、引当、履歴を正確に扱う必要がある |
| 会計 | RDB | 取引の正確性、監査、ロールバックが重要 |
| 権限管理 | RDB | ユーザー、ロール、組織、許可の関係を明確にしたい |

ただし、実際のシステムではNoSQLだけ、RDBだけで完結しないことも多い。たとえば、投稿本文やコメントはNoSQLに置き、決済や契約はRDBに置くように、用途で分ける判断もあり得る。

## 判断を間違えやすいところ

よくある失敗は、最初の作りやすさだけで選んでしまうことだ。

たとえば、Firestoreでプロトタイプを作ると、ログイン、データ保存、リアルタイム更新まで短時間で形にしやすい。しかし、後から「部署別の集計をしたい」「複数条件で検索したい」「更新履歴を監査したい」「注文と在庫を必ず同時に更新したい」となると、最初のデータ構造が足かせになることがある。

逆に、最初からRDBで厳密に設計しすぎると、まだ仕様が固まっていないサービスでは変更が重くなることがある。SNSや実験的なアプリでは、最初から業務システムのような正規化を求めすぎると、プロトタイプの速度が落ちる。

つまり、見るべきなのは「どちらが正しいか」ではなく、「今の用途で何を守る必要があるか」だ。

## 選定時に確認したいこと

データベースを選ぶ前に、少なくとも次の点は確認したい。

- 業務上、絶対にズレてはいけないデータか
- 複数データをまとめて更新する必要があるか
- 失敗時にロールバックが必要か
- 複雑なJOINや集計が必要か
- データ構造は頻繁に変わるか
- 読み取り回数や書き込み回数はどれくらいか
- 権限管理や監査ログが必要か
- 将来、別のDBへ移行する可能性があるか
- 無料枠や従量課金で、読み書き回数が問題にならないか

特にクラウドDBでは、保存容量だけでなく、読み取り回数、書き込み回数、インデックス、転送量、バックアップ、ログもコストに関係する。無料枠で小さく動いたとしても、本番利用のアクセス数でどうなるかは別に確認したい。

## 自分ならどう使い分けるか

個人開発で、SNS、メモ、チャット、画像投稿のようなものを素早く試すなら、FirebaseやFirestoreは候補になる。ログイン、データ保存、ストレージまで一気に試しやすく、初期の検証速度を上げやすい。

一方で、業務システム、契約、請求、在庫、会計、権限管理のように、データの正しさが問題になるものでは、最初からPostgreSQLのようなRDBを候補にする。Supabaseのようなサービスを使えば、個人開発でもPostgreSQLを比較的扱いやすい形で使える。

判断の基準は、作りやすさだけではない。あとから「正しいデータだった」と説明できる必要があるかどうかを見る。説明責任や監査が必要な領域では、RDBの方が説明しやすい場面が多い。

## まとめ

NoSQLとRDBは、優劣ではなく用途で選ぶ。

NoSQLは、柔軟なデータ構造、リアルタイム更新、素早いプロトタイプに向いている。一方で、RDBは、整合性、トランザクション、JOIN、監査、業務上の正しさを扱いやすい。

データベース選定では、最初に動くかだけでなく、後から正しさを説明できるか、変更に耐えられるか、運用コストを読めるかを確認したい。

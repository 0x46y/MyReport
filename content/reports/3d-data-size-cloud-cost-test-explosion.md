3Dデータの自動化では、実装そのものよりも、データ量、計算量、例外ケース、テストケースの組み合わせが問題になることがある。

Webアプリのフォーム入力やCSV処理のような感覚で見積もると、3Dデータではすぐに前提が崩れる。ファイルが大きい。読み込みに時間がかかる。メモリを使う。変換結果の正しさを判定しづらい。壊れたデータの壊れ方が毎回違う。

この記事では、3Dデータ自動化で事前に見ておきたい容量、クラウドコスト、処理時間、テストケース爆発について整理する。

## 3Dデータは単純なテキストより重くなりやすい

3Dデータは、点、エッジ、面、三角形、曲面、ソリッド、属性、アセンブリ構造など、多くの情報を持つ。

STLのようなメッシュ形式でも、三角形数が増えればファイルサイズはすぐに大きくなる。B-RepやSTEPのような形式では、曲線、曲面、トポロジー、属性、参照関係なども絡む。さらに、アセンブリになれば部品点数、階層、外部参照も増える。

テキスト処理なら1行ずつ読んで変換できる場合も多いが、3Dデータでは全体形状を読み込まないと判断できない処理が多い。

- 面が閉じているか
- 自由エッジがあるか
- ソリッドとして成立しているか
- 体積が異常でないか
- 面の向きが反転していないか
- 部品同士が干渉していないか
- 単位系やスケールが合っているか

これらは、単純な文字列変換より重い。

## クラウドに載せると転送量と保存コストも問題になる

3Dデータをクラウドで処理する場合、計算コストだけでなく、アップロード、ダウンロード、保存、キャッシュ、ログ、プレビュー画像なども考える必要がある。

たとえば大量のCADデータをクラウドで変換する場合、次のようなコストが発生する。

| 項目 | 見る理由 |
| --- | --- |
| アップロード容量 | 大きなCADファイルを毎回送ると時間がかかる |
| ダウンロード容量 | 変換後ファイルやプレビューを返すコストがある |
| ストレージ容量 | 元データ、変換後データ、中間データ、ログを保存する |
| メモリ使用量 | 形状解析やメッシュ化で大きなメモリを使う |
| CPU/GPU時間 | メッシュ生成、修復、干渉チェック、体積計算が重い |
| キャッシュ設計 | 同じデータを何度も変換しない工夫が必要 |
| セキュリティ | 製造データは機密性が高く、外部送信できない場合がある |

クラウドでやれば楽になる部分もあるが、製造業の3Dデータでは機密性の問題も大きい。社外にデータを出せない、特定リージョンに置けない、ログにファイル名や形状情報を残せない、といった制約もあり得る。

そのため、クラウド処理にするか、ローカル処理にするか、オンプレミスにするかは、技術だけでなくデータ管理方針にも依存する。

## 変換は成功・失敗の判定が難しい

3Dデータ変換では、ファイルが出力できたから成功とは限らない。

たとえば、STLを読み込んでSTEPに変換できたとしても、次のような問題が残ることがある。

- 見た目は同じだが、ソリッドではなくサーフェス集合になっている
- 自由エッジが残っている
- 小さな穴がある
- 面の向きが一部反転している
- 体積が異常に大きい、またはゼロに近い
- 単位がmmではなくinchとして扱われている
- アセンブリ構造が失われている
- 部品名や属性情報が落ちている

このため、変換後には品質確認が必要になる。見た目のスクリーンショットだけでは足りない。

確認項目としては、少なくとも次のようなものが考えられる。

| 確認項目 | 意味 |
| --- | --- |
| 読み込み可否 | 出力ファイルを対象CADやビューアーで開けるか |
| 面数・エッジ数 | 変換前後で極端な増減がないか |
| 自由エッジ | 閉じていない境界が残っていないか |
| ソリッド性 | 体積を持つ閉じた形状として扱えるか |
| 体積 | 異常に膨張・収縮していないか |
| バウンディングボックス | スケールや座標が壊れていないか |
| 属性 | 部品名、材質、PMIなど必要な情報が残っているか |

## テストケースは組み合わせ爆発しやすい

3D自動化で特に厄介なのは、テストケースが組み合わせ爆発しやすいことだ。

単純に「CATIAからSTEPへ変換する」だけでも、実際には多くの軸がある。

- CADソフトの種類
- CADのバージョン
- ライセンス構成
- 入力形式
- 出力形式
- 部品かアセンブリか
- 単位系
- 座標系
- 面数
- 曲面の複雑さ
- 履歴の有無
- 参照リンクの有無
- 欠損や破損の有無
- 期待する品質基準

これらの組み合わせをすべて試すのは現実的ではない。

だからこそ、最初に「代表ケース」「危険ケース」「失敗してもよいケース」を分ける必要がある。

| ケース | 目的 |
| --- | --- |
| 代表ケース | よく使う正常データで期待通り動くか確認する |
| 大型ケース | ファイルサイズ、メモリ、処理時間の限界を見る |
| 壊れたケース | 自由エッジ、穴、反転面などの異常に耐えられるか見る |
| 境界ケース | 小さすぎる形状、大きすぎる形状、薄板、複雑曲面を見る |
| ライセンス差分ケース | 出力形式やAPIが環境で変わらないか見る |

## 企業側が見落としやすいこと

発注側から見ると、3D自動化は「人がやっている変換を自動化するだけ」に見えることがある。

しかし実際には、人が暗黙に判断していることが多い。

- 変換後に見た目を確認している
- 面が欠けていたら別形式で出し直している
- 失敗しそうなデータを経験で避けている
- ライセンスのある端末でだけ処理している
- 大きすぎるデータは手作業で軽量化している
- 取引先ごとに出力設定を変えている

これを自動化するなら、その暗黙知を条件として書き出す必要がある。

「いつも人がやっているから簡単」ではなく、「人が見て判断していた部分を、プログラムでどう判定するか」を決める必要がある。

## PoCで確認したいこと

3Dデータ自動化では、最初から全機能を作るより、PoCで次の点を確認した方がよい。

| 確認項目 | 理由 |
| --- | --- |
| 対象データの代表性 | たまたま簡単なデータだけで成功していないか確認する |
| 最大ファイルサイズ | メモリや処理時間の上限を見る |
| 失敗パターン | どの壊れ方で止まるか確認する |
| 品質判定 | 成功と失敗をどう区別するか決める |
| 処理時間 | 業務上待てる時間か確認する |
| 実行環境 | ローカル、オンプレ、クラウドのどこで動かすか決める |
| ログ | 失敗時に再現できる情報を残せるか確認する |

PoCの目的は、完璧なシステムを作ることではない。できること、難しいこと、コストがかかること、代替案が必要なことを早めに見つけることだ。

## まとめ

3Dデータの自動化では、容量、計算量、クラウドコスト、品質判定、テストケースの組み合わせ爆発を先に見る必要がある。

ファイルが出力できたから成功ではない。見た目が同じだから十分とも限らない。ソリッドとして成立しているか、必要な属性が残っているか、スケールや単位が壊れていないか、失敗時に原因を追えるかが重要になる。

特に製造業では、3Dデータが大きく、機密性も高く、例外ケースも多い。だからこそ、最初に対象データ、品質基準、実行環境、失敗時の扱いを整理してから進める方がよい。

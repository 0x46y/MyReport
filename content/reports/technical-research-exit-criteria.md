結論から言うと、技術調査の目的は成功を証明することだけではない。

どの条件なら進められるのか、どの条件なら止めるべきなのかを明確にすることも重要な成果である。

小さな検証（PoC）や技術調査では、最初からすべての制約が分かっているとは限らない。調べる中で、ライセンス、性能、費用、運用、外部依存、データ品質、責任分界点の問題が見つかることがある。そのときに撤退条件がないと、「もう少し調べれば解決するはず」と考えて、調査だけが長引きやすい。

この記事では、技術調査で撤退条件を決める意味を整理する。

PoC、撤退条件、外部依存、責任分界点、非機能要件などの用語は [開発・クラウド・CADの記事で使う用語メモ](/notes/system-development-terms) に短く整理している。

## 撤退条件は失敗を決めるものではない

撤退条件という言葉は、少し後ろ向きに聞こえるかもしれない。

しかし、撤退条件は「失敗したら終わり」という意味ではない。むしろ、どこまで調べたら判断できるのかを先に決めるための条件だ。

たとえば、次のような状態を事前に決めておく。

| 判断 | 意味 |
| --- | --- |
| 継続 | 主要なリスクが許容範囲で、次の検証や本開発に進める |
| 条件付き継続 | 追加確認や制約対応が終われば進められる |
| 保留 | 判断材料が不足しており、今は進めない |
| 撤退 | 重要な制約があり、現時点では進めない |
| 別案検討 | 現在の案ではなく、別の方法に切り替える |

ここで大事なのは、撤退を「負け」と扱わないことだ。技術的な制約、費用、ライセンス、運用負荷が見えたなら、それは調査の成果である。

## なぜ撤退条件が必要なのか

技術調査では、「もう少し調べればできるかもしれない」という状態になりやすい。

もちろん、調査を続けることで解決策が見つかることはある。一方で、明確な判断基準がないまま続けると、次のような問題が起きる。

- 調査期間だけが伸びる
- 費用や工数が膨らむ
- 関係者の期待値がずれる
- 誰が継続判断をするのか曖昧になる
- 本来検討すべき別案に移れない
- 調査が終わっていないのに、実装が進んだように見える

技術調査で重要なのは、可能性を探ることと同時に、見込みが薄い方法を早めに外すことだ。

撤退条件があると、感情や根性論ではなく、あらかじめ決めた基準で判断しやすくなる。

## よくある長期化パターン

技術調査が長期化するときは、いくつかのパターンがある。

| パターン | 起きること |
| --- | --- |
| 成功条件だけ決めている | できた場合の話はあるが、できない場合の判断がない |
| 期限が曖昧 | いつまで調べるのか決まっていない |
| 費用上限がない | 追加調査や外部サービス利用が積み上がる |
| 外部依存を軽く見る | ベンダー製品、API、ライセンスの制約で止まる |
| 判断者が不明 | エンジニア、営業、発注側の誰が止めるのか決まっていない |
| 代替案がない | 現在の案に固執して、別の方法を検討できない |

特に危ないのは、「技術的にはあと少しに見える」状態だ。

画面は出ている。サンプルデータでは動く。デモでは見せられる。だが、本番データ、ライセンス、性能、運用、監査、費用を入れると成立しない。この状態で撤退条件がないと、PoCと本番導入の間で止まりやすい。

この点は [小さな検証（PoC）が成功しても本番導入できない理由](/reports/poc-success-production-adoption-risks) にも整理している。

## 撤退条件として見たい観点

撤退条件は、単に「できなかったら止める」では弱い。

何がどの水準を超えたら止めるのか、または別案に切り替えるのかを決めたい。

| 観点 | 撤退条件の例 |
| --- | --- |
| 技術 | 必須APIが存在しない、必要なデータを取得できない |
| 性能 | 実データで処理時間が許容値を超える |
| 費用 | 月額費用やAPI料金が上限を超える |
| ライセンス | 商用利用、自動化、大量処理が許可されない |
| 運用 | 障害時に原因を追えない、復旧手順を作れない |
| データ | 本番データの品質が低く、自動処理できない |
| セキュリティ | 機密情報や個人情報を外部サービスへ送れない |
| 保守 | 担当者が変わると再現できない |
| 組織 | 業務上の正しさを判断できる人がいない |

撤退条件は、技術だけで決めるものではない。業務、費用、運用、契約、責任範囲も含めて判断する。

## 成功条件と撤退条件を対で書く

技術調査では、成功条件だけでなく撤退条件も同じタイミングで書いた方がよい。

たとえば、次のように整理できる。

| 調査項目 | 成功条件 | 撤退条件 |
| --- | --- | --- |
| 外部API連携 | 必要な項目をAPIから取得できる | 必須項目がAPIで提供されていない |
| ファイル変換 | 実データの主要パターンを変換できる | 変換時に必要情報が失われる |
| AI活用 | 誤りを検出し、人が確認できる | 正解判定がなく、誤回答を検出できない |
| クラウド運用 | 月額費用と運用負荷が許容範囲 | 転送料やAPI料金が上限を超える |
| DB選定 | 必要な整合性と性能を満たす | トランザクションや権限管理が足りない |
| 本番環境 | 本番相当のURLと権限で動く | ローカルでは動くが本番ランタイムで動かない |

こうしておくと、「どこまでできたか」だけでなく、「どこで止めるか」も説明しやすい。

## 撤退条件を決める例

たとえば、PDFをAIで分類する仕組みを検討するとする。

この場合、最初に次のように整理しておくと判断しやすい。

| 項目 | 内容 |
| --- | --- |
| 目的 | PDFを内容別に分類し、確認作業の時間を減らす |
| 成功条件 | 50件のテストデータで90%以上を正しく分類できる |
| 撤退条件 | 50件のテストデータで80%未満、または月額費用が3万円を超える |
| 保留条件 | 精度は出るが、機密情報の扱いや利用規約の確認が終わっていない |
| 別案 | AIで分類候補だけを出し、最終判断は人が行う |

このように書いておくと、「AIで分類できるか」だけでなく、「どの水準なら業務で使えると判断するか」「どの水準なら完全自動化をやめるか」を話しやすくなる。

特にAIを使う場合、100%の精度を前提にすると本番導入が難しくなることがある。そのため、完全自動化を目指すのか、候補提示にとどめるのか、人による確認を残すのかを早めに分けておきたい。

## 撤退ではなく別案に切り替える

撤退条件に該当したからといって、プロジェクト全体をやめるとは限らない。

多くの場合は、現在の案をやめて別案に切り替える判断になる。

たとえば次のような切り替えがある。

- 完全自動化ではなく、人の確認を残す
- リアルタイム処理ではなく、夜間バッチにする
- AI判断ではなく、AIは候補出しだけにする
- 外部API連携ではなく、CSV取り込みにする
- 高精度な3D変換ではなく、レビュー用の軽量表示にする
- クラウド処理ではなく、ローカル処理や社内環境で処理する
- 全社導入ではなく、対象部署や対象データを絞る

撤退条件は、開発を止めるためだけのものではない。現実的な別案に移るための基準でもある。

## 関係者と合意しておきたいこと

撤退条件は、エンジニアだけで決めても十分ではない。

業務上の優先順位、費用、納期、契約、利用者への影響が関係するため、発注側、営業側、運用側、開発側で合意しておきたい。

合意したいこと:

- 調査期間はいつまでか
- どの結果が出たら次に進むか
- どの結果が出たら保留や撤退にするか
- 追加調査が必要になった場合の費用はどう扱うか
- 代替案を誰が判断するか
- 既存データや業務判断は誰が確認するか
- 調査結果をどの形式で残すか

撤退条件を先に共有しておくと、途中で難しい制約が見つかったときにも、個人の感覚ではなく合意した基準に沿って話しやすくなる。

この点は [不確実性が高い開発では、合意形成と進め方を先にそろえたい](/reports/exploratory-development-contract-risk-agile) ともつながる。

## 技術調査の成果物として残す

技術調査の成果は、動くコードだけではない。

むしろ、次のような判断材料を残すことが重要になる。

- 調査した範囲
- 試した方法
- 成功した条件
- 失敗した条件
- 未確認の条件
- 採用しなかった案
- 撤退条件
- 次に進む場合の追加確認
- 別案に切り替える場合の候補

これらが残っていると、後から「なぜその判断をしたのか」を説明しやすい。

特に会社やチームで進める場合、調査した本人がいなくなっても、判断の前提が残ることに価値がある。

## 自分ならどう進めるか

自分が技術調査を進めるなら、最初に次の形で整理する。

1. 何を実現したいのか
2. 何が分かれば次に進めるのか
3. 何が分かったら止めるのか
4. どの条件なら別案に切り替えるのか
5. 誰が業務上の正しさを判断するのか
6. 費用と期間の上限はどこか

そのうえで、調査結果を次のように分類する。

| 判断 | 状態 |
| --- | --- |
| 採用候補 | 主要リスクが許容範囲で、次の検証へ進める |
| 条件付き採用 | 追加確認や制約対応が必要 |
| 保留 | 判断材料が足りない |
| 撤退 | 現在の案では重要条件を満たせない |
| 別案検討 | 目的は残すが、方法を変える |

この分類を使うと、技術調査を「調べました」で終わらせず、次の意思決定につなげやすい。

## まとめ

技術調査で撤退条件を決めることは、後ろ向きな作業ではない。

成功条件だけでなく、保留、撤退、別案検討の条件を決めておくことで、調査の長期化、費用増加、責任範囲の曖昧化を防ぎやすくなる。

技術調査の目的は、成功を証明することだけではない。成立条件と撤退条件を明確にし、次に進むべきか、止めるべきか、別案に切り替えるべきかを判断できる状態にすることだ。

撤退条件が明確であれば、難しい制約が見つかったときにも、関係者が同じ前提で判断しやすくなる。

結論から言うと、AIがコードを書けるようになっても、人間が確認すべきことは残る。

AIは実装、調査、要約、修正案の作成をかなり速くできる。一方で、情報が現在の実装と合っているか、ローカルと本番環境で同じように動くか、重い処理を何度も繰り返していないか、そもそも進む方向が妥当かは、人間が確認した方がよい場面が多い。

この記事では、AIコーディングを使うときに確認したいことを、技術調査とPoCの観点で整理する。

PoC、API、SDK、Edge Runtime、Node.js Runtime、キャッシュなどの用語は [開発・クラウド・CADの記事で使う用語メモ](/notes/system-development-terms) に短く整理している。

## AIが得意なことと、人間が確認したいこと

AIを使うと、実装速度はかなり上がる。既存コードの読み取り、テスト追加、エラーの調査、ドキュメント要約、サンプルコード作成は強い。

ただし、AIが出したコードや説明をそのまま採用する前に、次の観点を確認したい。

| 観点 | AIが得意なこと | 人間が確認したいこと |
| --- | --- | --- |
| 情報収集 | 記事やドキュメントの要約 | 情報が今のバージョンや実装に合っているか |
| 実装 | コード生成、修正案、リファクタ | その方針でよいか、運用に耐えるか |
| 環境構築 | 手順の整理、依存関係の推定 | 実際のソースコードや設定と一致しているか |
| デバッグ | エラー原因の候補出し | 重い検証を無駄に繰り返していないか |
| 本番化 | 設定例やデプロイ案の提示 | 本番環境、権限、監視、費用に合っているか |
| 方針判断 | 候補の列挙 | どこで止めるか、どこで別案に切り替えるか |

つまり、人間の役割は「全部を手で書くこと」から、「AIが進める方向と前提条件を確認すること」に寄っていくと考えている。

## 記事やAIの回答だけでなく、ソースコードも確認する

AIの回答や技術記事は便利だが、それだけでは分からないことがある。

以前、Hugging Faceで公開されているLLMや生成系AIを試そうとしたとき、ブラウザ上のデモ自体は普通に動作していた。作者がデモとして動く状態を公開しているため、画面上では正常に試せる。

一方で、自分の環境で最新のコードをクローンし、モデルをダウンロードして動かそうとすると、記事や説明だけでは環境構築や呼び出し方法が分かりにくいことがあった。GitHubのソースコードを確認すると、pipで依存関係をまとめて入れられるようになっていたり、READMEでは目立っていない実際の読み込み方が実装から分かったりした。

ここで重要なのは、「ブラウザデモが壊れていた」という話ではない。デモとして動いているものと、最新コードを手元で動かすことは別の作業だということだ。

確認したいこと:

- ブラウザデモとGitHub上のコードは同じ前提か
- README、記事、サンプルコード、実装のどれが最新に近いか
- 依存関係は手動で集める必要があるのか、パッケージ管理で入るのか
- 関数の呼び出し方やモデルの読み込み方が変わっていないか
- Issues、release notes、commit履歴に重要な変更がないか

公開されているソースコードを確認できるなら、AIにそのコードを読ませて、実際の呼び出し方、依存関係、設定ファイル、テストを見てもらう価値がある。技術調査では、記事やAIの回答だけでなく、生のソースコードも一次情報として扱いたい。

## 実行環境の差分を見る

AIが生成したコードがローカルで動いても、本番環境で同じように動くとは限らない。

たとえば、Cloudflare Workers、Vercel、AWS Lambda、Firebase Functionsのような環境では、Node.js API、ファイルシステム、タイムアウト、環境変数、リクエストボディ、時刻、ヘッダーの扱いが変わることがある。

特にWebhook、OAuth、Stripe連携、ファイルアップロード、AI API呼び出しは、ローカルだけで判断すると危ない。

確認したいこと:

- Node.js Runtimeで動くのか、Edge Runtimeで動くのか
- ローカルと本番で環境変数やシークレットの渡し方が同じか
- Webhookのraw bodyや署名検証が本番でも通るか
- タイムアウトやメモリ制限に収まるか
- ログを見て原因を追えるか

この点は [ローカルで動くことと本番環境で動くことは違う](/reports/local-production-edge-runtime-differences) にも整理している。AIがコードを書いたかどうかに関係なく、本番環境との差分は早めに確認したい。

## 非機能要件をAIに明示する

AIに「この処理を直して」と頼むと、AIは目的達成を優先して進めることがある。

しかし、実際の開発では、処理時間、検証時間、再実行コスト、保存容量、ログ量、失敗時の復旧方法も重要になる。これらは非機能要件だ。

以前、OCCTとWebAssemblyを使って3Dデータ変換や形状処理を試したとき、1回の変換に20分から40分かかることがあった。AIはデバッグのたびに重い変換を最初から繰り返そうとすることがあり、そのままでは検証時間が大きく膨らむ。

この場合に必要だったのは、変換処理そのものを何度も走らせることではなく、途中まで成功した結果を保存し、そこから再開できるチェックポイントを作ることだった。

確認したいこと:

- 1回の検証に何分かかるか
- 同じ重い処理を何度も繰り返していないか
- 中間成果物を保存して再利用できるか
- 小さなサンプルデータで先に検証できるか
- 失敗時にどこから再開できるか
- キャッシュやログをどう残すか

AIに任せる場合でも、「この処理は重いので毎回走らせない」「まず小さいデータで確認する」「中間ファイルを保存する」「失敗したら再開地点を作る」といった制約を明示した方がよい。

## 開発時間もコストとして見る

AIを使うとコードを書く時間は短くなる。しかし、実行時間や検証時間まで自動的に短くなるわけではない。

たとえば、AIが毎回同じ重いビルド、変換、テスト、データ生成を走らせると、実装自体は速くても、検証に時間がかかり続ける。クラウド上で実行する場合は、その時間が料金にもつながる。

確認したいこと:

- どの処理が一番時間を使っているか
- 失敗時に最初からやり直していないか
- テストを軽いものと重いものに分けているか
- AIに渡すタスクを小さく分けているか
- 実行コストがクラウド料金に反映されないか

AI時代でも、時間と費用の感覚は重要だ。特に3D、AIモデル、動画、音声、巨大データの処理では、試行回数そのものがコストになる。

## 進む方向を人間が確認する

AIは、与えられた目標に向かって進むことが得意だ。

一方で、前提条件そのものが間違っている場合や、別のライブラリを選ぶべき場合、機能を削るべき場合、いったん撤退すべき場合には、人間が方向修正する必要がある。

たとえば、実装は進んでいるが、そもそも選んだライブラリが本番環境に対応していない。コードは動くが、無料枠を超えたときの費用が合わない。ローカルでは成功するが、本番の権限やWebhookで詰まる。このような場合、コードを増やすより、方針を戻して確認した方がよい。

確認したいこと:

- このライブラリやサービスを選ぶ理由はまだ有効か
- 最初に決めた前提が崩れていないか
- 実装を続けるより、別案を試す方が早くないか
- ここまで作ったものを捨てる判断が必要ではないか
- 撤退条件を決めているか

AIに実装を任せるほど、人間は「もっと速く書く」より「この方向で進んでよいか」を見る役割が強くなる。

## AIが見えにくい情報を見る

AIが苦手なのは、論理的に考えることそのものではない。むしろ、AIが見えにくいのは、契約書、仕様書、工程表、READMEには書かれていない情報だ。

以前、工事現場に関する話として聞いたことがある。トラックや機材が足りない状況では、工程表や契約条件だけを見れば、「リスケする」「追加工数を申請する」「別途調達する」という判断になる。これは合理的な判断だ。

しかし実際の現場では、過去の取引関係や信頼関係をもとに、別の業者へ一時的な協力を依頼し、後日別の形で調整することもあるという。

ここで重要なのは、「AIには絶対にできない」という話ではない。過去の取引履歴、契約条件、担当者同士の関係、許可されている交渉範囲、責任分界点までデータとして与えれば、AIも候補を出せる可能性はある。

ただし、実際にその依頼をするか、相手との関係に借りを作るか、失敗したときに誰が責任を持つかは、人間や組織が判断する必要がある。

ソフトウェア開発でも同じことがある。特定ベンダーとの関係、運用担当者の負荷、社内ルール、過去の障害対応履歴、監査上の注意点などは、仕様書やREADMEだけでは見えないことがある。AIは技術的に可能な案を出せるが、組織として採用してよいかは別途確認したい。

確認したいこと:

- 契約書や仕様書に書かれていない暗黙知がないか
- 過去の取引関係や信頼関係が判断に影響しないか
- AIに見えていない制約や関係者がいないか
- その判断の責任を誰が持つのか
- 法令、契約、社内ルールに反しないか
- 「できる方法」ではなく「やってよい方法」か

AIに任せる範囲を広げるほど、技術的に可能かだけでなく、組織として採用してよいかを確認する必要がある。

## AIに渡すとよい制約

AIに依頼するときは、実装内容だけでなく制約も渡した方がよい。

たとえば次のような情報を先に伝える。

- 使える実行時間
- 使える料金の上限
- 触ってよいファイル
- 実行してよいコマンド
- 重い処理を繰り返してよいか
- 本番環境で使うランタイム
- 必ず確認したいテスト
- 使ってはいけないサービスやライブラリ

AIは制約が明確なほど、現実的な提案をしやすくなる。逆に、制約を伝えないと、技術的には可能だが、時間、費用、運用に合わない案を出すことがある。

## 人間が確認したいチェックリスト

AIコーディングを使うときは、少なくとも次の点を確認したい。

| 観点 | 確認すること |
| --- | --- |
| 情報源 | AIの回答だけでなく、公式ドキュメント、README、ソースコードも見る |
| バージョン | 記事や回答が今のライブラリに合っているか |
| 実行環境 | ローカル、本番、Edge、Node.jsの差分を見る |
| 依存関係 | pip、npm、モデルファイル、外部サービスの前提を見る |
| 非機能要件 | 処理時間、メモリ、保存容量、転送料を見る |
| 検証効率 | キャッシュ、中間成果物、チェックポイントを作る |
| コスト | 無料枠、API料金、クラウド実行時間を見る |
| 方向性 | 続行、方針転換、撤退の判断をする |
| 組織判断 | 暗黙知、信頼関係、責任分界点、社内ルールを見る |

この表は、AIを疑うためのものではない。AIを安全に速く使うための確認項目だ。

## まとめ

AIがコードを書く時代でも、人間が確認すべきことは残る。

AIは実装速度を上げる。一方で、情報源の鮮度、実際のソースコード、実行環境、非機能要件、検証時間、クラウド費用、撤退条件、責任分界点は人間が確認した方がよい。

重要なのは、AIを使うか使わないかではなく、AIに何を任せ、人間が何を確認するかを分けることだ。

コードを書く能力だけでなく、成立条件を確認し、方向修正し、運用できる形に近づける能力が、今後さらに重要になると考える。

結論から言うと、小さな検証（PoC）が成功したことと、本番導入できることは別である。

小さな検証では「技術的に動くか」を早く確認できる。一方で、本番導入では、費用、運用、権限、監視、保守、責任分界点、利用規約、既存システムとの接続まで含めて成立する必要がある。

この記事では、小さな検証（PoC）が成功しても本番導入で止まりやすい理由を整理する。

PoC、非機能要件、責任分界点、Webhook、OAuth、egress などの用語は [開発・クラウド・CADの記事で使う用語メモ](/notes/system-development-terms) に短く整理している。

## 小さな検証（PoC）で確認できることと、確認しきれないこと

小さな検証（PoC）は有効な進め方だ。最初から大きく作るより、小さく試した方が、技術的な見込みや制約を早く確認できる。

ただし、PoCで確認できることには限界がある。

| 観点 | PoCで確認しやすいこと | 本番導入前に追加で確認したいこと |
| --- | --- | --- |
| 技術 | 主要機能が動くか | 本番環境、負荷、権限、障害時にも動くか |
| 費用 | 小さい利用量で試せるか | 月額、年額、転送料、運用人件費が成立するか |
| データ | サンプルデータで処理できるか | 本番データ量、移行、バックアップ、権限管理に耐えるか |
| 運用 | 手元で実行できるか | 誰が監視し、誰が復旧し、誰が問い合わせを受けるか |
| 組織 | 関係者に見せられるか | 契約、法務、セキュリティ、社内ルールに合うか |

PoCの目的は、単に「動きました」と示すことではない。本番導入時に問題になりそうな制約を早めに見つけることだ。

## 技術的成功と業務的成功は違う

PoCでは、画面が動く、APIが返る、AIモデルが応答する、ファイル変換が成功する、といった技術的な成功を確認しやすい。

しかし、業務で使うには別の条件が必要になる。

確認したいこと:

- 現場の作業時間を本当に減らせるか
- 入力データの揺れに耐えられるか
- 例外や失敗時の手順があるか
- 担当者が変わっても運用できるか
- 利用者が結果を判断できるか
- 業務上の正しさを誰が確認するか

たとえば、AIで文書を要約できたとしても、その要約を誰が確認するのか、誤りがあったとき誰が責任を持つのか、社外に出してよいデータなのかは別に確認する必要がある。

「技術的に可能」と「業務として採用してよい」は分けて考えたい。

## 本番環境で動くか

PoCがローカル環境で動いても、本番環境で同じように動くとは限らない。

特に、Webhook、OAuth、Stripe連携、ファイルアップロード、Edge Runtime、Node.js Runtime、クラウド関数、メール送信のような処理は、環境差分で止まりやすい。

確認したいこと:

- 本番相当のURL、HTTPS、ドメインで動くか
- 環境変数やシークレットを安全に渡せるか
- 本番のランタイムで利用ライブラリが動くか
- タイムアウト、メモリ、ファイルサイズ制限に収まるか
- ログを見て原因を追えるか
- ローカルではなく、デプロイ後の環境で確認したか

この点は [ローカルで動いたのに本番で動かないのはなぜか](/reports/local-production-edge-runtime-differences) にも整理している。

PoCの段階で本番相当の環境を一度通しておくと、後から「動くと思っていたが本番では動かない」というリスクを減らせる。

## コストが成立するか

PoCでは利用量が小さいため、費用が見えにくい。

無料枠や小さな従量課金で動いたとしても、本番導入後にユーザー数、データ量、API呼び出し回数、転送料、ストレージ容量が増えると、費用構造が変わることがある。

確認したいこと:

- 月額費用はいくらか
- 年額ではいくらか
- 利用量が10倍、100倍になった場合どうなるか
- 外向き転送やAPI呼び出しに課金されるか
- 監視、ログ、バックアップ、サポート費用を含めているか
- 無料枠を超えたときに止めるのか、課金して継続するのか

この点は [無料枠で検証するときに確認したいこと](/reports/free-tier-verification-checklist) や [クラウドのランニングコストとベンダーロックインを見落とさないために](/reports/cloud-running-cost-vendor-lockin) ともつながる。

PoCで重要なのは、無料で動いたことではなく、費用が増えたときの判断材料を残すことだ。

## データ運用できるか

PoCでは、少量のサンプルデータで動くことが多い。しかし本番では、データ量、権限、更新頻度、バックアップ、復旧、移行が問題になる。

確認したいこと:

- 本番データ量で性能が足りるか
- 個人情報や機密情報を扱うか
- 誰が読み書きできるか
- 誤更新や削除から復旧できるか
- バックアップとリストアを確認したか
- データを別サービスへ移行できるか
- NoSQL、RDB、オブジェクトストレージの選択が用途に合っているか

たとえば、SNSやチャットのように柔軟なデータ構造を扱うならNoSQLが向く場合がある。一方で、受発注、在庫、会計のように整合性や履歴が重要な用途ではRDBを検討したい。

この点は [NoSQLとRDBはどの条件で選ぶのか](/reports/nosql-rdb-selection-conditions) や [個人開発でDBをどう選ぶか: SQLite / Prisma / Supabase / Firebase](/reports/personal-development-database-selection) にも整理している。

## 監視と障害対応ができるか

PoCでは、作った本人が手元で動かして確認することが多い。本番では、作った本人が常に見ているとは限らない。

確認したいこと:

- エラーが起きたときに通知されるか
- ログを見て原因を追えるか
- 誰が一次対応するか
- 復旧手順があるか
- 外部サービス障害時の代替手段があるか
- 利用者への告知や問い合わせ対応を誰が行うか

本番導入とは、コードを置くことだけではない。障害が起きたときに、誰が気づき、誰が判断し、誰が戻すかまで含めて考える必要がある。

## 責任分界点が明確か

PoCは小さく始めるため、責任範囲が曖昧でも進められることがある。しかし本番導入では、責任分界点を曖昧にしたまま進めると後で揉めやすい。

確認したいこと:

- どこまでが開発側の責任か
- どこからが利用者側、運用側、発注側の責任か
- 外部サービスの障害は誰が判断するか
- データの正しさを誰が確認するか
- セキュリティや法務の確認は誰が行うか
- 完成条件と受け入れ条件は明確か

特にAI、外部API、SaaS、クラウド、CADソフト、決済サービスを使う場合、自分たちで直せる範囲と、外部サービス側に依存する範囲を分ける必要がある。

この点は [AIがコードを書く時代に人間は何を確認するのか](/reports/ai-code-generation-human-checkpoints) とも関係する。AIや外部サービスが候補を出せても、採用してよいか、責任を持てるかは人間と組織が確認する必要がある。

## 担当者が変わっても保守できるか

PoCは、作った本人の理解に依存しやすい。

しかし本番導入後は、担当者が異動する、外部ベンダーが変わる、作った人が退職する、数か月後に別の人が修正する、ということが起こる。

確認したいこと:

- READMEや運用手順があるか
- 環境構築を再現できるか
- 依存関係やバージョンが記録されているか
- デプロイ手順が属人化していないか
- テストやサンプルデータがあるか
- 外部サービスの管理画面や権限が共有されているか

PoCの段階では、多少手順が雑でも動かせる。しかし本番導入するなら、作った本人以外でも最低限追える状態にしておきたい。

## PoC段階で残したい判断材料

PoCを本番導入につなげるには、結果だけでなく判断材料を残す必要がある。

残したいこと:

- 何を確認するためのPoCだったか
- どの条件では成功したか
- どの条件では未確認か
- 採用した案と採用しなかった案
- 本番導入前に追加で確認すること
- 月額費用と運用負荷の見積もり
- 撤退条件

「PoCは成功しました」だけでは、本番導入の判断材料として足りない。

むしろ次のように残した方がよい。

| 判断 | 意味 |
| --- | --- |
| 採用候補 | 主要なリスクを確認でき、本番導入の検討に進める |
| 条件付き採用 | 追加検証や費用確認が終われば進められる |
| 保留 | 技術的には動いたが、運用や費用の判断材料が足りない |
| 不採用 | 重要な制約があり、現状では採用しない |
| 別案検討 | 方針自体を変えた方がよい |

PoCの価値は、成功したか失敗したかだけではない。どの条件なら進められ、どの条件なら進めないかを明確にすることにある。

## 本番導入前のチェックリスト

PoCから本番導入へ進む前に、少なくとも次の点は確認したい。

| 観点 | 確認すること |
| --- | --- |
| 目的 | 何を解決するための導入か |
| 利用者 | 誰が使い、誰が判断するか |
| 環境 | 本番相当のURL、権限、ランタイムで動くか |
| データ | 本番データ量、権限、バックアップに耐えるか |
| 費用 | 月額、年額、転送料、サポート費用を見たか |
| 運用 | 監視、ログ、障害対応、問い合わせ対応があるか |
| セキュリティ | シークレット、個人情報、権限管理を確認したか |
| 法務 | 利用規約、商用利用、契約条件に合うか |
| 保守 | 担当者が変わっても再現できるか |
| 判断 | 採用、保留、不採用、別案検討の条件を残したか |

このチェックリストは、PoCを止めるためのものではない。本番導入で起きる問題を早めに見つけるためのものだ。

## まとめ

小さな検証（PoC）が成功したことと、本番導入できることは別である。

PoCでは、技術的に動くかを早く確認できる。一方で、本番導入では、費用、運用、データ、監視、保守、責任分界点、利用規約、既存システムとの接続まで含めて判断する必要がある。

PoCの目的は、技術的成功を示すことだけではない。本番導入に進める条件と、進めない理由を早めに見つけることだ。

本番導入が難しいと分かった場合は、どの時点で保留や撤退を判断するかも重要になる。この点は、別の記事で整理したい。
